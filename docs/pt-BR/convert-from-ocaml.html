Convertendo de OCaml

Tal qual Reason é apenas uma sintaxe alternativa a OCaml, converter um projeto de OCaml é simples e não requer alterações semânticas. No entanto, há algumas configurações de compilação e mudanças gerais requeridas.

OCamlBuild -> Rebuild

Reason vem com um substituto compatível com o ocambuild chamado rebuild, que automaticamente compilará qualquer arquivo Reason junto de seus arquivos OCaml, sem configuração adicional. Isso permite que você adicione arquivos Reason ao seu projeto OCaml existente gradualmente. Onde seu script se referir ao ocamlbuild, apenas substitua por rebuild.

Makefile

Se o seu sistema de compilação executa comandos de compilação explícita, então a maneira mais fácil de usar Reason com ocamlopt/ocamlc é adicionando as seguintes flags para cada etapa de compilação:

# intf-suffix diz ao compilador onde procurar arquivos de interface correspondentes
ocamlopt -pp refmt -intf-suffix rei -impl myFile.re
ocamlopt -pp refmt -intf myFile.rei

Se você estiver usando o ocamlbuild sem rebuild, adicione o seguinte ao seu arquivo _tags, mas isso provavelmente não será suficiente porque ocamlc/ocamlopt vai precisar das flags -intf/-impl/-intf-suffix:

<**/*.{re,.rei}>: package(reason), syntax(utf8)

Correção da Sintaxe do Construtor

O código Reason convertido pode acoplar [@implicit_arity] a construtores de variantes, da seguinte forma: C 1 2 [@implicit_arity]. Isso é devido ao fato de que o OCaml tem sintaxe ambígua, onde um construtor com múltiplos argumentos espera argumento em forma de tupla. Por isso, em tempo de análise, não sabemos se C (1, 2) em OCaml deve ser traduzido para C (1, 2) ou C 1 2 em Reason. Por padrão, converteremos para C 1 2 [@implicit_arity], que avisa o compilador que isso pode ser tanto C 1 2 quanto C (1, 2).

Para prevenir a geração de [@implicit_arity], é possível fornecer o argumento --assume-explicit-arity ao refmt. Isso força o formatador a gerar C 1 2 ao invés de C 1 2 [@implicit_arity].

Entretanto, como C 1 2 requer múltiplos argumentos, a compilação pode falhar caso a expressão seja na verdade um construtor com uma única tupla como argumento (ex: Some). Nós já temos algumas regras internas de exceções que cobrem os construtores comuns que requerem uma única tupla como argumento, de forma tal que eles serão convertidos corretamente (ex: Some (1, 2) será convertido em Some (1, 2) ao invés de Some 1 2, que não compila).

Para fornecer sua própria lista de exceções, crie um arquivo separado em linhas que contém todos os construtores (sem prefixos de módulo) em seu projeto que esperam uma única tupla como argumento, e use o argumento --heuristics-file <nome-do-arquivo> para avisar ao refmt que todos os construtores listados no arquivo serão tratados como construtores com uma única tupla como argumento:

> cat heuristicas.txt
  TupleConstructor
  And
  Or

> cat teste.ml

type tm =
  TupleConstructor of (int * int)
| MultiArgumentsConstructor of int * int
let x = TupleConstructor(1, 2)
let y = MultiArgumentsConstructor(1, 2)
module test = struct
  type a = | And of (int * int) | Or of (int * int)
end;;
let a = Test.And (1, 2)
let b = Test.Or (1, 2)
let c = Some (1, 2)

Assim, será assumido que apenas os construtores cujos identificadores foram listados aceitam tuplas, ao invés de múltiplos argumentos.

> refmt --heuristics-file \
    ./heuristicas.txt --assume-explicit-arity \
    --parse ml --print re teste.ml

type tm = 
  | TupleConstructor((int, int))
  | MultipleArgumentsConstructor(int, int);

let x = TupleConstructor((1, 2));

let y = MultiArgumentsConstructor(1, 2);

module Test = {
  type a = 
    | And((int, int))
    | Or((int, int));
};

let a = Test.And((1, 2));

let b = Test.Or((1, 2));

let c = Some((1, 2));
