Módulos

Básico

Módulos são como mini arquivos! Eles contém definições de tipo, let bindings, módulos aninhados etc.

Criação

Para criar um módulo, use a palavra-chave module. O nome do módulo deve começar com uma letra maiúscula. Qualquer coisa que você possa colocar num arquivo .re, você pode colocar dentro do bloco {} da definição de um módulo.

module School = {
  type profession = Teacher | Director;

  let person1 = Teacher;
  let getProfession = (person) =>
    switch (person) {
    | Teacher => "A teacher"
    | Director => "A director"
    };
};

O conteúdo de um módulo (incluindo tipos!) pode ser acessado assim como o de um record, usando a notação .. Isso demonstra a utilidade dos módulos para namespacing.

let anotherPerson: School.profession = School.Teacher;
print_endline(School.getProfession(anotherPerson)); /* "A teacher" */

Módulos aninhados também funcionam.

module MyModule = {
  module NestedModule = {
    let message = "hello";
  };
};

let message = MyModule.NestedModule.message;

Abrindo/acessando um módulo (open)

Se referir constantemente a um valor/tipo em um módulo pode ser tedioso. Nós podemos abrir a definição de um módulo e fazer referência ao seu conteúdo sem usar o nome do módulo como prefixo. Duas maneiras:

Abertura local.

let message =
  School.(
    switch (person1) {
    | Teacher => "Hello teacher!"
    | Director => "Hello director!"
    }
  );

Abertura global. Use isto com moderação, já que mesmo isso sendo uma conveniência, gera um custo na facilidade de entendimento do código:

open School;
let anotherPerson: profession = Teacher;
printProfession(anotherPerson);

Estendendo módulos

Usar include em um módulo estaticamente "espalha" o conteúdo do módulo em um novo, assim, muitas vezes, cumpre o papel feito por "herança" ou "mixin"s.

module BaseComponent = {
  let defaultGreeting = "Hello";
  let getAudience = (~excited) => excited ? "world!" : "world";
};

module ActualComponent = {
  /* o conteúdo é copiado */
  include BaseComponent;
  /* sobrepõe BaseComponent.defaultGreeting */
  let defaultGreeting = "Hey";
  let render = () => defaultGreeting ++ " " ++ getAudience(~excited=true);
};

Nota: abrir (open) e incluir (include) são muito diferentes! O primeiro traz todo o conteúdo de um módulo para o escopo atual, de modo que você não precisa referir-se a um valor prefixando-o com o nome do módulo toda vez. O segundo transcreve a definição de um módulo estaticamente, e então também faz um open.

To arquivo .re é um módulo

Um arquivo OCaml/Reason é mapeado para um módulo; o interessante é que isso traz alguma expressividade que em outras linguagens exigiria geração de código. O arquivo react.re implicitamente forma um módulo React, que pode ser enxergado por código em outros arquivos.

/* arquivoA.re. Isso geralmente é compilado para o módulo ArquivoA abaixo */
let a = 1;
let b = 2;

/* arquivoB.re */
/* Mapeia a implementação de arquivoA para uma nova API */
let alpha = FileA.a;
let beta = FileA.b;

Esse trecho inventado expressa copiar um arquivo:

/* arquivoA.re. Isso geralmente é compilado para o módulo ArquivoA abaixo */
let a = 1;
let b = 2;

/* arquivoB.re */
/* compila para exatamente o mesmo conteúdo de arquivoA.re sem perda de desempenho em execução! */
include ArquivoA;

Assinaturas

O tipo de um módulo é chamado de "assinatura", e pode ser escrito explicitamente. Se um módulo é como um arquivo (de implementação) .re, então a assinatura de um módulo é como um arquivo (de interface) .rei.

Criação

Para criar uma assinatura, use a palavra-chave module type. A assinatura precisa começar com uma letra maiúscula. Qualquer coisa que você possa colocar num arquivo .rei, você pode colocar dentro do bloco {} da definição de uma assinatura.

/* Partindo do exemplo da seção anterior */
module type EstablishmentType = {
  type profession;
  let getProfession: profession => string;
};

Uma assinatura define uma lista de requerimentos que um módulo precisa satisfazer para corresponder a essa assinatura. Estes requerimentos são da forma:

  • let x: int;``let requer um binding nomeado x, do tipo int.
  • type t = someType; requer que um campo de tipo t seja igual a someType.
  • type t; requer um campo de tipo t, mas sem obrigar qualquer imposição no atual tipo concreto t. Nós podemos usar t em outros trechos da assinatura para descrever relacionamentos, e.g. let makePair: t => (t, t) porém sem assumir, por exemplo, que t seja um int. Isto nos compele a usar grandes habilidades de abstração.

Para ilustrar as várias declarações de tipos, considere a assinatura EstablishmentType acima, que requer que um módulo:

  • Declare um tipo com nome profession.
  • Inclua uma função que tenha um valor do tipo profession e retorne uma string.

Observação:

Módulos com o tipo EstablishmentType podem conter mais campos do que aqueles que a assinatura declara, tal como o módulo School na seção anterior (se nós escolhêssemos atribuir a ele o tipo EstablishmentType. Caso contrário, School exporia todos os campos). Isso efetivamente torna o campo person1 um detalhe de implementação! Entidades externas não podem acessá-lo, dado que não está presente na assinatura; a assinatura restringiu o que outros podem acessar.

O tipo EstablishmentType.profession é abstrato: não possui um tipo concreto; ele está dizendo "não me importa qual seja o tipo real, somente que este é usado como entrada de getProfession". Isto é útil para fazer vários módulos se ajustarem à mesma interface:

module Company: EstablishmentType = {
  type profession = CEO | Designer | Engineer | ...;

  let getProfession = (person) => ...;
  let person1 = ...;
  let person2 = ...;
};

Também é útil para esconder o tipo subjacente como um detalhe de implementação do qual outros não podem depender. Se você perguntar qual o é tipo de Company.profession, ao invés de expor a variante, vai somente ser respondido como "é Company.profession".

Estendendo assinaturas de módulos

Assim como os próprios módulos, assinaturas de módulos também podem ser estendidas através de include module type of Foo:

module type BaseComponent = {
  let defaultGreeting: string;
  let getAudience: (~excited: bool) => string;
};

module type ActualComponent = {
  /* a assinatura é copiada */
  include (module type of BaseComponent);
  let render: unit => string;
};

Todo arquivo .rei é uma assinatura

Similar a como um arquivo react.re implicitamente define um módulo React, um arquivo react.rei implicitamente define uma assinatura para React. Se react.rei não é fornecido, a assinatura de react.re é assumida como o padrão, com todos os campos do módulo expostos. Por não poder conter arquivos de implementação, arquivos .rei também são usados no ecossistema para documentar a API pública dos módulos correspondentes.

/* arquivo react.re (implementação. Compila para módulo React) */
type state = int;
let render = (str) => str;

/* arquivo react.rei (interface. Compila para assinatura do módulo React) */
type state = int;
let render: str => str;

Funções de Módulos (funtores)

Módulos também podem ser passados para funções! Seria o equivalente a passar um arquivo como uma entidade de primeira classe. Entretanto, módulos estão em um "nível" diferente da linguagem em relação a outros conceitos comuns, portanto não podemos passá-los para funções comuns. Ao invés disso, nós passamos eles para funções especiais chamadas "funtores".

A sintaxe para definir e utilizar funtores é bem parecida com a sintaxe para definir e utilizar funções comuns. As principais diferenças são:

  • Funtores usam a palavra-chave module ao invés de let.
  • Funtores recebem módulos como argumentos e retornam um módulo.
  • Funtores exigem anotação de argumentos.
  • Funtores precisam iniciar com letra maiúscula (assim como módulos/assinaturas).

Segue um funtor MakeSet de exemplo, que recebe um módulo do tipo Comparable e retorna um novo conjunto que pode conter tais itens comparáveis.

module type Comparable = {
  type t;
  let equal: (t, t) => bool;
};

module MakeSet = (Item: Comparable) => {
  /* vamos usar uma lista como nossa (tosca) estrutura de dados de base */
  type backingType = list(Item.t);
  let empty = [];
  let add = (currentSet: backingType, newItem: Item.t) : backingType =>
    /* se o item existe */
    if (List.exists((x) => Item.equal(x, newItem), currentSet)) {
      currentSet /* retorna o mesmo conjunto imutável (uma lista na verdade) */
    } else {
      [
        newItem,
        ...currentSet /* prefixa ao conjunto e retorna-o */
      ]
    };
};

Funtores podem ser aplicados usando a sintaxe de aplicação de funções. no trecho abaixo, nós estamos criando um conjunto, cujos itens são pares de inteiros.

module IntPair = {
  type t = (int, int);
  let equal = ((x1, y1), (x2, y2)) => x1 == x2 && y1 == y2;
  let create = (x, y) => (x, y);
};

/* IntPair conforma-se à assinatura Comparable exigida por MakeSet */
module SetOfIntPairs = MakeSet(IntPair);

Tipos de funções de módulos

Assim como tipos dos módulos, tipos dos funtores também agem para restringir o que podemos assumir sobre funtores. A sintaxe para tipos dos funtores é consistente com aquela de tipos das funções, porém com tipos iniciando com letra maiúscula para representar as assinaturas de módulos que o funtor aceita como argumentos e valor de retorno. No exemplo anterior, nós estamos expondo o tipo subjacente do conjunto; dando a MakeSet uma assinatura de funtor, nós podemos esconder a estrutura de dados subjacente!

module type Comparable = ...

module type MakeSetType = (Item: Comparable) => {
  type backingType;
  let empty: backingType;
  let add: (backingType, Item.t) => backingType;
};

module MakeSet: MakeSetType = (Item: Comparable) => {
  ...
};

Desvantagens

Módulos e funtores estão em um "nível" diferente da linguagem em relação ao resto (funções, let bindings, estruturas de dados, etc.). Por exemplo, você não consegue passá-los facilmente para uma tupla ou um record. Use-os judiciosamente! Muitas das vezes, apenas um record ou uma função são o suficiente.

module School = {...};

/* isto lhe dará um erro de sintaxe! */
let schools = (School, School);
