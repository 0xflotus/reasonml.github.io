Record

Registros são como objetos JavaScript mas são

  • mais leves
  • imutáveis por padrão
  • fixos em nomes e tipos
  • muito rápidos
  • uma tipagem um pouco mais rígida

Uso

Tipos (obrigatório):

type person = {
  age: int,
  name: string
};

Valores (isso vai ser inferido para ser do tipo person):

let me = {
  age: 5,
  name: "Big Reason"
};

Acesso (a familiar notação do ponto):

let name = me.name;

Records Precisam de Definições Explícitas

Se você apenas escreve {age: 5, name: "Baby Reason"} sem uma declaração explícita em algum lugar acima, o sistema de tipos irá cuspir um erro. Se a definição do tipo reside em outro arquivo, você precisará indicar explicitamente qual arquivo é:

/* School.re */

type person = {age: int, name: string};

/* example.re */

let me: School.person = {age: 20, name: "Big Reason"};
/* or */
let me = School.{age: 20, name: "Big Reason"};
/* or */
let me = {School.age: 20, name: "Big Reason"};

Qualquer um dos três exemplos acima diz que "a definição deste registro é encontrada no arquivo School". A primeira com anotação de tipo normal é preferida.

Atualizações Imutáveis

Novos registros podem ser criados a partir de registros antigos com o operador de propagação .... O registro original não será modificado.

let meNextYear = {...me, age: me.age + 1};

Esta atualização é muito eficiente! Experimente alguns em nosso playground para ver como os registros são compilados.

Nota: propagação não pode adicionar novos campos, uma vez que a forma de um registro é fixada pelo seu tipo.

Atualizações Mutáveis

Campos de registros podem opcionalmente ser mutáveis. Isso permite que você atualize esses campos localmente com o operador =.

type person = {
  name: string,
  mutable age: int
};
let baby = {name: "Baby Reason", age: 5};
baby.age = baby.age + 1; /* alter `baby`. Happy birthday! */

Atalho de sintaxe

Para reduzir a redundância, fornecemos punning para os tipos e valores de um record. "Punning" refere-se ao atalho de sintaxe que você pode usar quando o nome de um campo corresponde ao nome do seu valor/tipo:

type horsePower = {power: int, metric: bool};

let metric = true;
let someHorsePower = {power: 10, metric};
/* same as the value {power: 10, metric: metric}; */

type car = {name: string, horsePower};
/* same as the type {name: string, horsePower: horsePower}; */

Observe que não há punning para um único campo de registro! {foo} não faz o que você espera (é um bloco que retorna o valor foo).

Dicas & Truques

Interoperabilidade com Javascript

Se você está trabalhando com JavaScript, a sintaxe e operações com registro vão lhe parecer familiares, e você pode até se sentir tentado a interoperar com JS convertendo um objeto JS em um registro, e vice e versa. Não há problema, no entanto, temos uma forma ainda melhor sem overhead de conversão! Clique aqui para ver sobre Reason objects. Aqui está um exemplo:

type payload = {. "name": string};

[@bs.module "myAjaxLibrary"] external sendQuery : payload => unit = "sendQuery";

sendQuery({"name": "Reason"});

Observe o ponto na definição de tipo. Isso é uma notação de tipo de objeto e não tem nada a ver com um record! Objetos serão descritos em uma seção posterior.

Tipos de registro são encontrados pelo nome do campo

Com records, você não pode dizer "Eu gostaria que essa função tivesse qualquer tipo de record, desde que ele tenha o campo age". O seguinte exemplo funciona, porém não como desejado:

type person = {age: int, name: string};
type monster = {age: int, hasTentacles: bool};

let getAge = (entity) => entity.age;

A última linha da função vai inferir que o parametro entity precisa ser do tipo monster. A última linha desse código falha:

let kraken = {age: 9999, hasTentacles: true};
let me = {age: 5, name: "Baby Reason"};

getAge(kraken);
getAge(me);

O sistema de tipos irá reclamar que me é um person, e a função getAge somente funciona com monster. Se você precisar de tal capacidade, use objetos Reason mencionados na seção anterior.

Decisões de Design

Depois de ler as restrições nas seções anteriores e se você tem conhecimento de linguagens dinâmicas, você deve estar se perguntando por que alguém se estressaria em usar um record em vez de usar um objeto diretamente, tal que o primeiro precisa ser explícito e não permite records diferentes com o mesmo nome de campo a serem passados para a mesma função, etc.

  1. A verdade é que a maioria das vezes em seu aplicativo, forma do seus dados na verdade é fixa, e se não for, ela pode potencialmente ser melhor representada como uma combinação de variante (introduzido em seguida) + records*.
  2. Record, desde que seus campos sejam fixos, é compilado para uma matriz com acessos de índice de matriz em vez de objeto JS (experimente no playground!). No nativo, ele compila para basicamente uma região de memória onde um campo de acesso é apenas um acesso único + um acesso atual, também conhecido como 2 instruções de assembly. Os bons velhos tempos onde o pessoal media em nanosegundos...
  3. Finalmente, uma vez que um tipo de record é resolvido através do encontro daquela declaração de único tipo explícito (chamamos esta "tipo nominal"), as mensagens de tipo erro acabam sendo melhores do que a contraparte ("tipos estruturais" como para tuplas). Isto torna a refatoração mais fácil; mudar o campos de um record do tipo permite naturalmente que o compilador saiba que ainda é o mesmo record, apenas usado menos em alguns lugares. Caso contrário, sob tipos estruturais, pode ser difícil dizer se q definição ou o uso está errado.

* E não estamos apenas encontrando desculpas! Objetos Reason suportam esses recursos.
