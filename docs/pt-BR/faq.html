Perguntas Frequentes

Eu devo aprender primeiro Reason ou OCaml?

Não precisa escolher! Reason e OCaml compartilham exatamente a mesma semântica (por exemplo, como o código executa). Apenas a sintaxe difere. Carregue Reason-tools aonde for e poderá converter livremente entre as duas sintaxes. Um tutorial de Reason é um tutorial de OCaml, e vice-versa. No terminal, você pode ter esses aliases:

# converte código de ocaml para reason
alias mlre="pbpaste | refmt --parse ml --print re --interface false | pbcopy"
# converte código de reason para ocaml
alias reml="pbpaste | refmt --parse re --print ml --interface false | pbcopy"

Eles pegarão seu código na área de transferência (do macOS), o converter, e colocá-o de volta na área de transferência! Troque pbpaste/pbcopy pelas funções de área de transferência do seu sistema.

Não estou certo do que fazer com Reason

Nós compilamos para JS muito bem. Pense no que você faria normalmente com JavaScript puro; tente portar/escrever isso em Reason + BuckleScript! Nós recomendamos tentar fazer projetos concretos, voltados ao usuário final (por exemplo uma ferramenta para linha de comando) ao invés de projetos a nível de infra (por exemplo um gerador de boilerplate). A última categoria requer expertise e compreensão de código Reason idiomático.

Qual a relação entre Reason, BuckleScript e OCaml?

Veja aqui. Reason é uma sintaxe para OCaml e suporta todas as suas features. BuckleScript compila código OCaml/Reason para JavaScript.

De onde vem todas essas funções print_endline, string_of_int?

Elas são da biblioteca padrão (stdlib), pré-abertas (opened) durante a compilação do seu arquivo. Por isso, você vê elas no escopo.

Posso ter uma função que imprima estruturas de dados arbitrárias?

Se você está compilando para JavaScript através do BuckleScript, você pode usar o console.log do JS através de Js.log. Se estiver compilando para nativo, você irá precisar de algo como ppx_show. Uma funcionalidade futura do OCaml (chamada modular implicit) irá resolver isso diretamente na linguagem.

Porque há um + para adicionar inteiros e +. para adicionar floats, etc.?

Veja aqui.

A biblioteca ___ funciona no Reason?

A maioria das bibliotecas JS deve fácilmente funcionar em Reason + BuckleScript. Do lado nativo, sabendo que o Reason é somente uma transformação de sintaxe: sim, elas irão funcionar com o Reason também. Porém o workflow nativo está atualmente em construção e precisa ser aperfeiçoado.

Qual é a história no lado servidor? Eu deveria compilar para nativo ou JavaScript e usar Node.Js?

Nós compilamos para nativo, mas o fluxo nativo está atualmente em desenvolvimento. Por enquanto, recomendamos compilar para JavaScript através do BuckleScript e usar as amarras (bindings) em reasonml-community ou de algum outro lugar.

Qual é a história assíncrona do BuckleScript?

Primeiro, se você não está interfaceando com nenhuma biblioteca que usa promessas (promises), você pode simplesmente usar chamadas de retorno (callbacks). Todos as entendem e tem bom desempenho.

Se você precisar amarrar a uma biblioteca JavaScript que usa promessas, ou comunicar com tal biblioteca, você pode usar as amarras para promessas do BuckleScript. Há também o potencial de ter algum açúcar sintático no futuro. No longo prazo, gostaríamos de implementar uma implementação de promessa com especificação devidamente compatível com OCaml/Reason, de forma tal que as otimizações do compilador sejam utilizadas.

Para uma solução Ocaml mais idiomática: no lado nativo OCaml, nós temos lwt e Async. Por enquanto não usamos eles na web, mas poderemos usá-los no futuro.

Qual é a história de testes (unitários)?

Alguns dos recursos da linguagem OCaml (não apenas tipos) podem ser capazes de adiar a necessidade de teste unitário. Enquanto isso, para compilação para JavaScript, estamos trabalhando em amarras para o Jest. Nós consideramos usar Jest para nativo também, caso o Jest seja escrito usando Reason no futuro (sem planos concreto ainda). OUnit é uma pequena biblioteca de teste nativo para agora.

O que é o arquivo .merlin na raiz do meu projeto?

É o arquivo de metadados para Merlin, o backend compartilhado de integração para autocompletar, pular-para-definição, etc. Para o Fluxo de trabalho JavaScript, o sistema de construção bsb gera o arquivo .merlin para você; Você não precisa incluir ele no sistema de controle de versão e não precisa modificá-lo manualmente.

Não vejo nenhum import ou require no meu arquivo; como funciona a resolução de módulos?

Reason/OCaml não requer que você escreva qualquer importação; módulos referidos em um arquivo são automaticamente buscados no projeto. Especificamente, um módulo Ola pede ao compilador para buscar pelo arquivo ola.re ou ola.ml (e seu arquivo de interface correspondente, ola.rei ou ola.mli, se disponível).

O nome do módulo é o nome do arquivo, com inicial em maiúscula. Ele tem que ser único por projeto; isso abstrai o sistema de arquivos e permite você mover arquivos à vontade sem mudar o código.

Some | None, contents, Array, List e todos esses são especiais? Da onde eles vêem?

Eles são definições normais de variantes/registros/módulos que vêm com a biblioteca padrão, aberta (opened) por padrão durante a compilação por conveniência.

O que um argumento precedido por um underscore (e.x. _ ou _foo) significa?

Digamos que você tenha List.map (fun item => 1) minhaLista. O argumento item não é usado e gerará um aviso do compilador. Usar fun _ => 1 indica que você está intencionalmente recebendo e ignorando o argumento, e portanto contorna o aviso. Alternativamente, fun _item => 1 tem o mesmo efeito, mas indica mais descritivamente o que você está ignorando.

O que é esse MyModule.t que continuo vendo?

Assumindo que MeuModulo seja o nome de um módulo, t é uma convenção da comunidade que indica "o tipo que representa aquele módulo, o que quer que isso signifique". Por exemplo, para o módulo Js.String, String.t é o tipo representando "uma string".

Por que existe uma Js_promise e também uma Js.Promise? E sobre o Js_array, Js_string e vários outros?

Como uma convenção, Js_foo é o modulo atual, e Js.Foo é apenas um apelido para isso. Eles são equivalentes. Prefira Js.Foo, por que este é o nome do módulo público e oficial.

Quando implicitos modulares & múltiplos núcleos & efeitos algebraicos estarão prontos?

Um dia eles estarão. Por enquanto, nos ajude a entregar mais código Reason! A popularidade ajudará a canalizar mais contribuições ao OCaml. Quanto menos o pessoal do Ocaml precisar se preocupar com questões triviais, mais eles podem focar em pesquisa e execução!

Por que o BuckleScript e bsb são tão rápidos? Como eu posso retardá-los?

BuckleScript é otimizado para desempenho ao longo de toda a pilha. Você pode tentar retardá-lo adicionando uma dúzia de camadas de indireção e metaprogramação. Tente:

  • Adicionando alguns loops infinitos aqui e ali.
  • Enfiar uma ferramenta de build na linha de produção.
  • Arrastar mais dependências para escrever um Olá Mundo.

Eu estou vendo um arquivo estranho .cmi/.cmx/.cmj/.cma referenciado num erro do compilador. De onde esses arquivos vêm?

A comunidade Ocaml frequentemente usa extensões de arquivo para distinguir entre tipos de código fonte, artefatos, e metadados, dependendo do alvo de seu build (nativo/bytecode/JavaScript). Segue uma visão geral de algumas das extensões de arquivo que você pode encontrar:

Código fonte

  • .ml: arquivo de código fonte OCaml
  • .mli: Arquivo de interface do OCaml; determina quais partes do arquivo .ml correspondente são visíveis ao mundo exterior
  • .re: Arquivo de código fonte Reason. Como .ml, mas para Reason
  • .rei: Arquivo de interface Reason. Como .mli, mas para Reason

Arquivos compilados

  • .cmi: Arquivo de interface compilada (.rei/mli)
  • .cmx: Arquivo objeto compilado para saída nativa (via ocamlopt)
  • .cmo: Arquivo objeto compilado para saída bytecode
  • .cmj: Arquivo objeto compilado para a web (via BuckleScript)
  • .cma: Arquivo de biblioteca para saída bytecode (equivalente a arquivos .a da linguagem C)
  • .cmxa: Arquivo de biblioteca para saída nativa
  • .cmt: Contém uma "Árvore tipada" – basicamente uma Árvore de Sintaxe Abstrata (AST) com todas as informações dos tipos
  • .cmti: Como um arquivo .cmt, mas para arquivos de interface
  • .cmxs: Plugin carregado dinâmicamente (para compilação nativa)
  • .o: Arquivo de objeto compilado nativamente
  • .out: Nome/extensão convencionados para a saída final produzida pelo ocamlc/ocamlopt (ex. ocamlc -o meuExecutavel.out)

Outros arquivos do Ecossistema OCaml

  • .mll: arquivo de definição do analisador léxico ocamllex
  • .mly: arquivo de definicão do gerador de parser ocamlyacc
  • .mldylib: Contém uma lista de caminhos de módulos que serão compilados e arquivados juntos para construir um alvo .cmxs correspondente (um plugin nativo)
  • .mliv: Arquivos específicos à biblioteca Batteries para alguns pré-processamentos personalizados.
  • .mllib: Biblioteca Ocaml (cma e cmxa)
  • .mlpack: Pacote Ocaml (cmo construído com a opção -pack)
  • .mlpp: Arquivos específicos à biblioteca Extlib para alguns pré-processamentos personalizados
  • .mltop: Arquivo top-level OCamlbuild, usado pelo OCamlbuild para gerar um arquivo .top
  • .odocl: Arquivo de documentação OCaml

Se alguma dessas explicações ainda for um pouco obscura, aqui estão as expansões de alguns dos termos usados acima: - AST: Árvore de Sintaxe Abstrata. A estrutura de dados advinda do código fonte, sobre a qual o compilador opera. - Ligação: O passo onde o compilador recebe vários arquivos intermediários compilados e os combina em um só. Ex: ligando A com B, porque o código fonte original de A continha referências a B. - Nativo: Construções que executam diretamente as instruções assembly da plataforma em questão. - Bytecode: Semelhante a código nativo, porém mais portável e menos performático - Arquivo objeto: Arquivos que possuem código de máquina porém não podem ser diretamente executados.

Você pode encontrar mais informações e contexto sobre muitas dessas extensões no site do OCaml e também na neste post de listagem de e-mail. Há também aprofundamentos na compilação nativa e bytecode que contém descrições mais detalhadas no manual OCaml.
