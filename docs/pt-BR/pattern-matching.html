Correspondência de Padrões!

Certifique-se de ter lido Variante antes.

Finalmente chegamos aqui! Pattern matching é um dos melhores recursos da linguagem. É como desestruturação, mas obtendo ainda mais auxílio do sistema de tipos.

Uso

Considere um variante:

type payload =
  | BadResult(int)
  | GoodResult(string)
  | NoResult;

Ao usar a expressão switch nele, você pode "desestruturá-lo":

let data = GoodResult("Product shipped!");

let message =
  switch data {
  | GoodResult(theMessage) => "Success! " ++ theMessage
  | BadResult(errorCode) => "Something's wrong. The error code is: " ++ string_of_int(errorCode)
  };

Note como desestruturamos data ao mesmo tempo que tratamos cada uma das variações. O switch no trecho acima vai gerar um alerta do compilador:

Warning 8: this pattern-matching is not exhaustive.
Here is an example of a value that is not matched:
NoResult

Não é ótimo? Enquanto checava o formato do nosso dado, o sistema de tipos nos alertou de uma variação não tratada. Este aspecto condicional é o que diferencia pattern matching da simples desestruturação. A maioria das estruturas de dados com um aspecto "se isso então aquilo" tomam proveito funcionalidade:

switch myList {
| [] => print_endline("Empty list")
| [a, ...theRest] => print_endline("list with the head value " ++ a)
};

switch myArray {
| [|1, 2|] => print_endline("This is an array with item 1 and 2")
| [||] => print_endline("This array has no element")
| _ => print_endline("This is an array")
};

O caso _ é um caso especial "pega-tudo" que permite que casos não considerados nas alternativas anteriores sejam processados neste ramo.

Você pode inclusive usar switch com strings, inteiros e outros tipos. Você pode até mesmo ter várias formas de dado direcionadas a um mesmo resultado!

let reply =
  switch message {
  | "Reason's pretty cool" => "Yep"
  | "good night" => "See ya!"
  | "hello" | "hi" | "heya" | "hey" => "hello to you too!"
  | _ => "Nice to meet you!"
  };

Combinado com outras estruturas de dados, pattern matching pode resultar em código extremamente conciso, verificado pelo compilador e com grande performance:

let message =
  switch data {
  | GoodResult(theMessage) => "Success! " ++ theMessage
  | BadResult(0 | 1 | 5) => "Something's wrong. It's a server side problem."
  | BadResult(errorCode) => "Unknown error occurred. Code: " ++ string_of_int(errorCode)
  | NoResult => "Things look fine"
  };

Observação: você pode passar somente literais (i.e. valores concretos) como um formato de dado, e não nomes definidos com let binding e outras coisas. O trecho a seguir não funciona como esperado:

let myMessage = "Hello";
switch greeting {
| myMessage => print_endline("Hi to you")
};

Ao contrário, ele estará correspondendo a qualquer string, e atribuindo esta ao nome myMessage naquele ramo do switch, o que não é o resultado almejado.

Cláusulas When (Quando)

Quando você realmente precisar de alguma lógica arbitrária num pattern match simples, você inserir algumas cláusulas when, que são basicamente açúcar sintático para if:

let message =
  switch data {
  | GoodResult(theMessage) => ...
  | BadResult(errorCode) when isServerError(errorCode) => ...
  | BadResult(errorCode) => ... /* otherwise */
  | NoResult => ...
  };

Match em Exceções

Se a função lançar uma exceção (serão apresentadas posteriormente), você pode também corresponder ao formato delas, em adição aos valores retornados normalmente pela função.

switch (List.find((i) => i === theItem, myItems)) {
| item => print_endline(item)
| exception Not_found => print_endline("No such item found!")
};

Dicas & Truques

Compacte suas checagens de formatos sempre que possível. Isto realmente lhe ajudará a remover bugs. Exemplo abaixo.

Não abuse muito do caso "pega-tudo" _. Isso previne o compilador de lhe avisar caso tenha esquecido de tratar uma variação de formato (checagem de exaustão/completude), o que seria de ajuda especialmente em casos de refatoração de código em que haja a adição de uma variação de formato. Tente usar _ somente em casos de possibilidades infinitas, e.g. string, inteiros, etc.

Segue uma série de exemplos, do pior para o melhor:

let optionBoolToJsBoolean = (opt) =>
  if (opt == None) {
    Js.false_
  } else {
    if (opt == Some(true)) {
      Js.true_
    } else {
      Js.false_
    }
  };

Isso é simplesmente tolo =). Vamos transformar isso em pattern matching:

let optionBoolToJsBoolean = (opt) =>
  switch opt {
  | None => Js.false_
  | Some(a) => a ? Js.true_ : Js.false_
  };

Um pouco melhor, mas ainda com aninhamento. Pattern matching permite você fazer o seguinte:

let optionBoolToJsBoolean = (opt) =>
  switch opt {
  | None => Js.false_
  | Some(true) => Js.true_
  | Some(false) => Js.false_
  };

A aparência é muito mais linear! Agora, podemos ser tentados a fazer isto:

let optionBoolToJsBoolean = (opt) =>
  switch opt {
  | Some(true) => Js.true_
  | _ => Js.false_
  };

Que é bem mais conciso, mas acaba com a completude da checagem mencionada acima. Isto é o melhor:

let optionBoolToJsBoolean = (opt) =>
  switch opt {
  | Some(true) => Js.true_
  | Some(false)
  | None => Js.false_
  };

Neste ponto é muito difícil cometer um erro nesse código! Em qualquer momento que você usaria um if-else com vários ramos, prefira pattern matching. É mais conciso e rápido também.

Decisões de Design

O notório problema fizzbuzz estranhamente deixa algumas pessoas deslocadas, em parte pela sua essência de deixar o programador travado na esperança de simplificar/unificar os poucos ramos condicionais na busca por elegância quando não há alternativa. Esperamos que você possa ver como geralmente a concisão de pattern matching em termos visuais nos permite superar essa trava de decisões, enquanto mantém os benefícios (e ainda mais, como visto) de uma sequencia de if-elses. Realmente não há nada errado em listar explicitamente todas as possibilidades; Pattern Matching corresponde à análise por casos da matemática, uma técnica válida para resolução de problemas, comprovadamente conveniente.

Usando o switch de Reason pela primeira vez talvez faça você sentir que esteve perdendo durante todos esses anos. Cuidado, isso pode estragar outras linguagens para você =).

Se você tentou refatorar uma lógica grande baseada em if-else s aninhados, você ocasionalmente descobrirá que é difícil manter a lógica correta. Por outro lado, pattern matching + tuplas são conceitualmente mapeadas para uma tabela, onde cada célula pode ser preenchida independentemente. Isto garante que toda vez que você precisa adicionar uma variação ao switch, você pode focar naquela e somente naquela célula da tabela, sem bagunçar outras células.

type animal = Dog | Cat | Bird;
let result = switch (isBig, myAnimal) {
| (true, Dog) => 1
| (true, Cat) => 2
| (true, Bird) => 3
| (false, Dog | Cat) => 4
| (false, Bird) => 5
};
