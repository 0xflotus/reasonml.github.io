Comparison to OCaml

如果你是一个OCaml的新手或者是一个正在学习OCaml的人，那么这个入门指导适合你。 但是别忘了 reason-tools可以自由的在OCaml和Reason语法间转换

注释

OCamlReason
(* OCaml (*nest*) *)/* Reason /*nest*/ */

运算符重命名

Reason有OCaml所有的中缀运算符，但是有几种运算符他们在表达上有区别。 在Reason中，结构相等写作==，引用（实际）相当写作 ===。 在Reason中，要想表达不相等，只要在等号字符前加上 !字符 (!= 是结构上不相等, 而!== 是引用不相等).

相等OCamlReason
结构x = yx == y
引用x == yx === y
不等OCamlReason
结构x <> yx != y
引用x != yx !== y

Local Scope

Reason的词法作用域和OCaml是完全相等的，但是在语法上更加类似于开发者熟知的‘‘块级作用域’’ 在Reason中，局部作用域由{}创建，在其中可能由let绑定和声明变量，并用;进行分割。 所有块计算到最后一行, 最后一行的分号是可选的。 只有当通过 ; 将多个项链在一起时, 才需要 {} 大括号。

<th>
  <p>
    Reason
  </p>
</th>

<td>
  <pre>

{ let msg = "Hello"; print_string(msg); let msg2 = "Goodbye"; print_string(msg2) };

OCaml 

let _ =
  let msg = "Hello" in
  print_string msg;
  let msg2 = "Goodbye" in
  print_string msg2

Reason的 {}移除了OCaml语法上的许多痛点。

  • 双分号被完全删除。
  • begin/end 被完全删除
  • 臭名昭著的命令式解析 问题 不见了
  • 模块体和局部作用域相统一

局部作用域 Vs 模块体

在Reason中，{} 块级作用域s和模块体内可以存在任何东西 通常可以在两个上下文之间剪切/粘贴代码。 在 OCaml 中, 两种上下文的语法非常不同。 本地作用域需要尾随 in, 但模块体不存在, 并且必须将某些命令性语句分配给 _ 或 (), 否则使用双 ;;。

在Reason中, 元组总是需要括号。

OCamlReason

fun ((a: int), (b: int)) -> a((a: int, b: int)) => a;

In Reason, record values resemble JavaScript, using : instead of =.

<h3>
  注释参数
</h3>

<p>
  在Reason和 OCaml 中, 参数通过类型批注 (与其他内容一样), 在追加 <code>: typeAnnotation </code> 后将它们包装在括号中。
</p>

<pre><code class="reason">(arg: argType) =&gt; returnValue;


<pre><code class="reason">(arg: argType) =&gt; (arg2: arg2Type) =&gt; returnValue;


<pre><code class="reason">(arg: argType, arg2: arg2Type) =&gt; returnValue;


<p>
  Reason和 OCaml 允许注释返回类型, 当使用 "let"的超级语法糖 形式。
</p>

<pre><code class="ocaml">(* OCaml *)

let myFunc (a:int) (b:int) :int * int = (a, b) let myFunc (a:int) (b:int) :int list = [1] let myFunc (a:int) (b:int) :int -> int = fun x -> x + a + b 

<pre><code class="reason">/* Reason */

let myFunc = (a: int, b: int) :(int, int) => (a, b); let myFunc = (a: int, b: int) :list(int) => [1]; let myFunc = (a: int, b: int): (int => int) => (x) => x + a + b; 

<p>
  因为我们在Reason中对所有函数都使用 <code> = &gt; </code>, 所以有一种情况是, 我们需要在一个本身是函数类型的返回类型周围添加额外的括号。
</p>

<h3>
  类型参数
</h3>

<h4>
  OCaml
</h4>

<p>
  OCaml 的类型应用程序 (认为 "泛型") 以相反的顺序应用。
</p>

<p>
  在 OCaml中, 会导致一些直观的后果。
</p>

<pre><code class="ocaml">let x: int list = [2]

type listOfListOfInts = int list list

(* Parsed as: *) type listOfListOfInts = (int list) list 

<p>
  当类型构造接受多个参数时，事情变得更加奇怪了。 多个参数需要括号和分号分割类型参数，但是这些括号不代表元组。 括号/逗号形式必须在<code>(int, string) tuple</code>这样的构造类型示例中给出。
</p>

<pre><code class="ocaml">type ('a, 'b) tuple = 'a * 'b

type listOfTuplesOfStringAndInt = (string, int) tuple list

(* Which is parsed as: *) type listOfTuplesOfStringAndInt = ((string, int) tuple) list

(* Which allows a list of (tuples of (string and int)) *) let tuples: listOfTuplesOfStringAndInt = [("asdf", 3)] 

<h4>
  Reason
</h4>

<p>
  总之, Reason将几乎所有的语法都统一为简单的 "函数应用程序" 样式, 这意味着类型参数遵循语法中其他地方所看到的相同的逗号分隔模式。 这将导致学习的语法模式更少。
</p>

<p>
  例如, 可以将 <code> list </code> 作为接受类型并返回新类型的类型的 "函数"。
</p>

<h2>
  作为类型参数的元组
</h2>

<p>
  因为 OCaml 使用括号和逗号来表示类型构造函数的多个参数, 所以当类型构造函数的一个参数本身是一个元组时, 它会混淆。 在 OCaml 中, 很难记住一个类型构造函数接受多个参数和一个类型构造函数接受一个恰好是元组的参数的区别。
</p>

<p>
  下面的示例显示了将 <em> 两个 </em> 类型参数传递给 <code> pair </code> 和一个恰好为元组的 <em> 单个 </em> 类型参数之间的区别。
</p>

<h2>
  变体
</h2>

<h5>
  OCaml
</h5>

<ul>
  <li>
    OCaml 已经期望构造函数参数类型以元组形式指定, 因此当一个构造函数需要一个 <em> 碰巧 </em> 为元组类型的单个参数时, 就会混淆。
  </li>
  <li>
    更让人费解的是, 构造函数 <em> 实际上 </em> 不接受元组, 但语法看起来像元组。
  </li>
  <li>
    有时, 用于实例化具有多个参数的构造函数的语法与构造变量的语法重叠, 该变量使用恰好是一个元组的单个参数, 因此它看起来像你支持了元组，事实上并没有。
  </li>
</ul>

<h5>
  Reason
</h5>

<ul>
  <li>
    变量构造函数类型应以逗号分隔列表的形式列出, 并使用括号来分组优先级 (与 <strong> "其他内容" </strong> 一样)。
  </li>
  <li>
    构造变量的实例 (如您所猜到的) 遵循函数应用程序样式 (逗号分隔列表)。
  </li>
  <li>
    元组 <strong> 总是 </strong> <em> 看起来 </em> 类似元组, 而任何看起来像元数组 <em> 的内容都是 </em> 元组。
  </li>
</ul>

<h2>
  各种改进
</h2>

<p>
  OCaml不强制在序列<code>(a;b;c;d)</code> 或者元组<code>(x,y)</code>外面加括号。 但是Reason总是用<code>{}</code>去包住序列或者let声明，而且Reason总是要求元组要有<code>()</code>，其他更直观的语法则不需要用括号去包裹。
</p>

</p>

<h3>
  注释转义
</h3>

<p>
  因为Reason使用C类型的注释，一些模糊的自定义前缀/中缀必须区分开来。 这些前缀/中缀操作符的规则类似于OCaml的语法，但是下面这些例外。
</p>

<p>
  特别的，如果如果前缀或者中缀字符的第一个字符是型号或者斜线，那么必须在前面加上反斜线。 在AST解析时反斜线会被去掉。 当打印时这些，反斜线会再次被加进去。
</p>

<table>
  <tr>
    <th>
      OCaml
    </th>
    
    <th>
      Reason
    </th>
  </tr>
  
  <tr>
    <td>
      <code>let (/*) a b = a + b</code>
    </td>
    
    <td>
      <code>let (/\*) a b =&gt; a + b;</code>
    </td>
  </tr>
  
  <tr>
    <td>
      <code>let x = 12 /-* 23 /-* 12</code>
    </td>
    
    <td>
      <code>let x = 12 /-* 23 /-* 12;</code>
    </td>
  </tr>
  
  <tr>
    <td>

  </tr>

    </td>
  </tr>
