JSX

想要在 Reason 來點 HTML 語法嗎？如果不想，快速略過這個章節，假裝什麼都沒看見！

Reason 支援 JSX 語法，跟 ReactJS 的比起來有些微小差異。 Reason JSX 沒有跟 ReatJS 綁在一起；只是轉譯成普通的函式呼叫：

注意 給 ReasonReact 讀者：這不是ReasonReact 轉換 JSX。詳情請見用法章節。

大寫字首標籤

<MyComponent foo={bar} />

變成

([@JSX] MyComponent.make(~foo=bar, ~children=[], ()));

非大寫字首標籤

<div foo={bar}> child1 child2 </div>;

變成

([@JSX] div(~foo=bar, ~children=[child1, child2], ()));

Children

<MyComponent> foo bar </MyComponent>

這是傳遞兩個元素 list 的語法，foo 和 bar 到 children 位置。這會解開成為 list 包含 foo 和 bar：

([@JSX] MyComponent.createElement(~children=[foo, bar], ()));

注意 這不是為了 ReasonReact 的轉換；ReasonReact 將最終的 list 傳換成 array。概念仍然相同。

所以自然地，<MyComponent> foo </MyComponent> 解糖成 ([@JSX] MyComponent.createElement(~children=[foo], ()))。 例如，無論你做什麼，放在 children 的參數都會被包裝成 list。 那如果不想要這樣呢？ 如果想要直接傳遞 foo 而不被包裝呢？

展開 Children

為了解決這個問題，我們引入

<MyComponent> ...foo </MyComponent>

傳遞 foo 且不會包裝在 list（或者 array，以 ReasonReact 案例）。意即，這會解糖成：

([@JSX] MyComponent.createElement(~children=foo, ()));

當你處理 foo 本身就是字串 list 且希望能夠繼續傳遞而不希望有額外的包裝時（也可能有型別錯誤），這個十分有幫助。 這也允許你傳遞任意資料結構到 children 位置（記得，JSX children 只是一般的 prop）：

<MyComponent> ...((theClassName) => <div className=theClassName />) </MyComponent>;
<MyForm> ...("Hello", "Submit") </MyForm>;

用法

請見 ReasonReact 有 JSX 應用程式範例，轉換上述呼叫成 ReasonReact 特定呼叫。

下列為 JSX 標籤，列出大多數的功能。

<MyComponent
  booleanAttribute={true}
  stringAttribute="string"
  intAttribute=1
  forcedOptional=?{Some("hello")}
  onClick={send(handleClick)}>
  <div> {ReasonReact.stringToElement("hello")} </div>
</MyComponent>

自 JS JSX 後的新方針

  • 屬性和 children 不需要用 {} 包起來，這裡為了容易學習將它顯示出來。一旦 refmt 你的檔案，某些會消失，某些會變成括弧。
  • 這裡不支援 JSX 展開 props： <Foo {...bar} />。儘管某些相關，這裡有展開 children，如上述： <Foo> ...baz </Foo>。
  • 雙關！

雙關

"雙關"指的是速寫語法，當標籤和值相同時使用。 例如，JavaScript 中，可以用 return {name} 寫法來取代 return {name: name}。

Reason JSX 支援雙關。 <input checked /> 是 <input checked=checked /> 的速寫。 當允許時，formatter 將幫你格式化成後者。 當有許多 props 時十分方便：

<MyComponent isLoading text onClick />

因此，在達到使用額外函式庫來避免傳遞 props 的極限之前，Reason JSX 可以塞的下更多的 props。

注意 這是從 ReactJS JSX 之後的新方針，原本沒有雙關。 ReactJS 的 <input checked /> 解糖成 <input checked=true />，為了符合遵循 DOM 的慣用語和向下相容性。

提示 & 技巧

給希望用到 JSX 優點的函式庫作者：上述的 [@JSX] 是個掛勾，供潛在的 ppx macros 發覺有個函式希望格式化成 JSX。 一旦發覺到函式，你可以將它轉換成其他表達式。

這樣，每個人都可以獲得 JSX 語法的好處而不需要引入特定的函式庫，例如 ReasonRact。

JSX 支援標籤式函式的功能：選項、明確地傳入選項和選項預設值。

設計決策

這樣設計 JSX 跟我們希望如何幫助語言發展有關。 請見這篇文章的 "What's the point?" 章節。

語言有巨集的能力 + 函式不可知的 JSX 語法允許每個函式庫可能地擁有 JSX 不須額外煩惱。 這樣，省去語意妥協（意即如何執行）便可加入一些視覺熟悉到底層 OCaml 語言。 Reason 其中一個重要目標是讓更多人可以獲得 OCaml 語言的美妙之處，同時免去在語法和格式上耗時的爭論。

* 你可能會想為什麼在 ReactJS 從來都不需要展開 children；ReactJS 用了某些特別的執行時期邏輯 + 特別語法轉換 + 多個參數偵測 & 標記來避免這些情況（請見這裡）。 這種動態用法讓型別系統偵測更加困難。 由於我們控制了整個語法和 ReasonReact，我們決定引入展開 children 來消除包裝與不包裝的歧義，不需要編譯 & 執行時期成本！
