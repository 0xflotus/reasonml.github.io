Record

Records 就像是 JavaScript 物件但是

  • 較輕量
  • 預設不可變
  • 欄位名稱及型別固定
  • 非常快
  • 些許嚴格的型別

用法

型別（必要）：

type person = {
  age: int,
  name: string
};

值（將會推論出 person 型別）：

let me = {
  age: 5,
  name: "Big Reason"
};

存取（熟悉的點表示法）：

let name = me.name;

Record 需要明確地定義

如果直接寫 {age: 5, name: "Baby Reason"} 沒有明確地在上方其他地方定義，型別系統會給你錯誤。 如果型別定義在其他檔案中，需要明確地指出哪個檔案：

/* School.re */

type person = {age: int, name: string};

/* example.re */

let me: School.person = {age: 20, name: "Big Reason"};
/* 或者 */
let me = School.{age: 20, name: "Big Reason"};
/* 或者 */
let me = {School.age: 20, name: "Big Reason"};

上述 3 種皆說明 "此 record 的定義可以在 School 檔案中找到"。偏好第一種，常見的型別標記。

修改不可變

透過 ... 擴散操作子（spread operator），可以從舊有的 record 產生出新的。舊有的 record 無法變動。

let meNextYear = {...me, age: me.age + 1};

這樣的修改效率很高！ 遊樂場試試看 record 如何編譯。

注意：擴散無法新增欄位，如同 record 外型是固定的。

修改可變

Record 欄位也可以選擇性可變。允許直接修改該欄位，透過 = 操作子。

type person = {
  name: string,
  mutable age: int
};
let baby = {name: "Baby Reason", age: 5};
baby.age = baby.age + 1; /* 修改 `baby`。生日快樂！ */

語法速記

為了減少冗餘，我們提供雙關給 record 的型別及其值。 雙關參照語法速記，當欄位名稱的值及型別與變數名稱相同時：

type horsePower = {power: int, metric: bool};

let metric = true;
let someHorsePower = {power: 10, metric};
/* 如同值 {power: 10, metric: metric}; */

type car = {name: string, horsePower};
/* 如同型別 {name: string, horsePower: horsePower}; */

注意，只有單一欄位的 record 並沒有雙關！ {foo} 跟你想像不一樣（這表示一個區塊，傳回 foo 的值）。

提示 & 技巧

與 JavaScript 交互使用

如果你用在 JavaScript 環境，record 語法& 操作應該感到熟悉。可能嘗試透過轉換 JS 物件與 record 進行交互使用。 這個沒問題，但是有更好的方式，省下轉換的功！ 請見這裡

Record 型別是從欄位找出

當使用 Records 時，你無法表示 "這個函式接受任意 record 型別，只要有 age 欄位就好"。下列方式可以運作，但不是你所期待：

type person = {age: int, name: string};
type monster = {age: int, hasTentacles: bool};

let getAge = (entity) => entity.age;

最後一行的函式推論其參數 entity 必須是 monster 型別。下列程式碼最後一行錯誤：

let kraken = {age: 9999, hasTentacles: true};
let me = {age: 5, name: "Baby Reason"};

getAge(kraken);
getAge(me);

型別系統抱怨 me 是個 person 型別，getAge 只接受 monster 型別。 If you need such capability, use Reason objects, described here.

設計決策

經過閱讀前面章節所提到的限制之後，如果你的背景來自動態語言，可能想知道為什麼要弄一個 record 而不直接用 object，前者需要明確型別且不允許同個函式接受相同欄位名稱的型別等。

  1. 事實上，大多數在你的應用程式中，實際上你的資料外型是固定的，如果不是固定的，可能最好結合變異（接下來會介紹）+ record *。
  2. Record，因為欄位是固定的，所以編譯成陣列並使用陣列索引存取，而非 JS object（遊樂場試試！）。 在原生中，基本上編譯出一塊記憶體區域，欄位存取只須一個欄位查找 + 一個實際存取，亦即 2 個組語命令。 回想過去人們用納秒（nanosecords）作測量...
  3. 最後，由於 record 型別是透過尋找單一特定型別定義（我們稱為 "nominal typing"），最後型別錯誤訊息比副本（"structural typing"，像是 tuples）來的好。 這使得重構更加容易；更改 record 型別的欄位自然地讓編譯器知道是同個型別，不會誤用其他地方。 否則，若在 structural typing，很難知道是定義方或者使用方錯誤。

* 不是為我們自己在找藉口！Reason objects 確實支援這些功能。
