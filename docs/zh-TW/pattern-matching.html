樣式匹配

請先確保你已經閱讀過變異。

終於到這了！樣式匹配是這個語言最棒的功能。像是解構，但更多來自型別系統的幫助。

用法

考慮下列變異：

type payload =
  | BadResult(int)
  | GoodResult(string)
  | NoResult;

當使用 switch 表達式，可以"解構"：

let data = GoodResult("產品送出！");

let message =
  switch (data) {
  | GoodResult(theMessage) => "成功！ " ++ theMessage
  | BadResult(errorCode) => "發生錯誤。 錯誤代碼：" ++ string_of_int(errorCode)
  };

注意當處理不同情況時，對 data 的解構。上述 switch 會得到編譯器警告：

Warning 8: this pattern-matching is not exhaustive.
Here is an example of a value that is not matched:
NoResult

很棒吧？ 當匹配資料外型時，型別系統會警告尚未處理的情況。 這種條件式的局面會進行樣式匹配而不是一般的解構。 大多數資料結構都帶有"如果是則.."的局面：

switch (myList) {
| [] => print_endline("空的 list")
| [a, ...theRest] => print_endline("list 的開頭值 " ++ a)
};

switch (myArray) {
| [|1, 2|] => print_endline("此陣列有項目 1 和 2")
| [||] => print_endline("此陣列沒有任何元素")
| _ => print_endline("這是個陣列")
};

_ 是個特殊失敗情況，允許所有條件都未匹配時進入此分支。

你甚至可以 switch 字串、int 和其他。更甚至有許多樣式得到相同結果！

let reply =
  switch (message) {
  | "Reason 蠻酷的" => "是啊"
  | "晚安" => "再會！"
  | "哈嘍" | "嗨" | "嘿呀" | "嘿" => "你好啊！"
  | _ => "很高興認識你！"
  };

與其他資料結構結合，樣式匹配可以產出非常簡潔、編譯器驗證、高效率的程式碼：

let message =
  switch (data) {
  | GoodResult(theMessage) => "成功！ " ++ theMessage
  | BadResult(0 | 1 | 5) => "發錯錯誤。 It's a server side problem."
  | BadResult(errorCode) => "發生未知錯誤。 錯誤代碼：" ++ string_of_int(errorCode)
  | NoResult => "看起來沒問題"
  };

注意：你只能傳遞字面上（如，具體的值）當作樣式，不能是 let-繫結的名字或其他。下列並不會如預期運作：

let myMessage = "哈嘍";
switch (greeting) {
| myMessage => print_endline("嗨你好")
};

反而，這會假設匹配任意字串，並將結果繫結到 switch 內的 myMessage 這個名稱，不是你所要的結果。

When 條件

如果真的需要任意邏輯，具有乾淨的樣式匹配，可以使用 when 條件，基本上是 if 語法糖：

let message =
  switch (data) {
  | GoodResult(theMessage) => ...
  | BadResult(errorCode) when isServerError(errorCode) => ...
  | BadResult(errorCode) => ... /* 否則 */
  | NoResult => ...
  };

匹配例外

如果函式會丟出例外（稍後說明），也可以匹配例外，另外也可以匹配函式傳回的結果值。

switch (List.find((i) => i === theItem, myItems)) {
| item => print_endline(item)
| exception Not_found => print_endline("找不到該項目！")
};

巢狀樣式

巢狀 | 如預期運作：

switch (student) {
| {name: "Jane" | "Joe"} => ...
| {name: "Bob", Job: Programmer({fullTime: Yes | Maybe})} => ...
};

樣式無所不在

任何一般"變數宣告"的地方都可以用樣式：

type leftOrRight =
  | Left(int)
  | Right(int);

let i = Left(1);

/* magic! */
let Left(v) | Right(v) = i;

提示 & 技巧

盡可能的讓樣式匹配扁平. 這是個真實的臭蟲去除劑。下列範例：

不要經常濫用特殊失敗情況 _。 這讓編譯器無法提醒你忘記處理某些情況（徹底檢查），特別在重構之後，加入了新的變異時，特別有幫助。 試著盡量使用 _ 於無限可能時，例如字串、int 等。

下列幾個範例，從糟糕到最佳：

let optionBoolToBool = opt =>
  if (opt == None) {
    false;
  } else if (opt == Some(true)) {
    true;
  } else {
    false;
  };

現在，只是缺乏思考 =)。試著轉成樣式匹配：

let optionBoolToBool = opt =>
  switch (opt) {
  | None => false
  | Some(a) => a ? true : false
  };

好一些，但仍然是巢狀。樣式匹配可以做到：

let optionBoolToBool = opt =>
  switch (opt) {
  | None => false
  | Some(true) => true
  | Some(false) => false
  };

看起來更加線性！現在，可能會試著：

let optionBoolToBool = opt =>
  switch (opt) {
  | Some(true) => true
  | _ => false
  };

更加簡潔，但是抹煞了上述的徹底檢查。這個最好：

let optionBoolToBool = opt =>
  switch (opt) {
  | Some((true | false) as trueOrFalse) => trueOrFalse
  | None => false
  };

現在這份程式碼幾乎很難犯錯！ 當有許多 if-else 分支時，偏好使用樣式匹配。 這個更加簡潔且高效。

請見其他範例，使用 switch + tuple。

設計決策

知名的 fizzbuzz 問題奇怪地絆倒某些人，部份原因是程式開發者的生性麻痺，希望尋找出不存在的單純/單一化條件分支。 希望你可以看見，樣式匹配一如往常的視覺簡潔讓我們克服決策麻痺，同時保有所有 if-else 分支的好處（更多即將看見）。 明確地列出所有可能性並沒有錯；樣式匹配類似數學的案例分析（case analysis），一種有效的解決問題技巧，證明非常方便。

第一次使用 Reason 的 switch 或許會讓你感到自己遺失多年。當心，這可能會讓你毀了其他語言 =)。

如果試著重構一個巨大、巢狀 if-else 邏輯，會發現非常難以讓邏輯正確。 換句話說，樣式匹配 + tuple 概念上映射成 2D 表格，每格都可以獨立滿足。 確保當你需要在 switch 加入新的案例，可以針對該表格的那一格，不會漏掉其他格。

type animal = Dog | Cat | Bird;
let result = switch (isBig, myAnimal) {
| (true, Dog) => 1
| (true, Cat) => 2
| (true, Bird) => 3
| (false, Dog | Cat) => 4
| (false, Bird) => 5
};
