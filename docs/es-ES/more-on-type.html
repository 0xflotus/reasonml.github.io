Más de tipo

¡Argumentos de tipos!

Los tipos pueden aceptar parámetros, similar a los genéricos en otros lenguajes. Es como si un tipo es una función que recibe argumentos y ¡devuelve un tipo nuevo! Los parámetros necesitan empezar con '.

El caso de uso de un tipo parametrizado es eliminar duplicaciones. Antes:

/* esto es una tupla de 3 items, explicado después */
type intCoordinates = (int, int, int);
type floatCoordinates = (float, float, float);

let buddy: intCoordinates = (10, 20, 20);

Después:

type coordinates('a) = ('a, 'a, 'a);

/* aplica las coordinadas a"function de tipo" y devuelve el tipo (int, int, int) */
type intCoordinatesAlias = coordinates(int);

let buddy: intCoordinatesAlias = (10, 20, 20);

/* o, mas comúnmente, escríbelo directamente en el código */
let buddy: coordinates(float) = (10.5, 20.5, 20.5);

En la práctica, los tipos se infieren por tí. Entonces la versión más sencilla de el ejemplo anterior sería:

let buddy = (10, 20, 20);

El sistema de tipos deduce que es un (int, int, int). Nada más es necesario para ser anotado.

Argumentos de tipos aparecen en todas partes.

/* inferido como `list(string)` */
let greetings = ["hola", "mundo", "como estas"];

Si los tipos no aceptan parámetros (es decir, si no tuviéramos "funciones de tipo"), la biblioteca estándar necesitaría definir los tipos listOfString, listOfInt, listOfTuplesOfInt, etc.

Los tipos pueden recibir más argumentos y ser componibles.

type result('a, 'b) =
  | Ok('a)
  | Error('b);

type myPayload = {data: string};

type myPayloadResults('errorType) = list(result(myPayload, 'errorType));

let payloadResults: myPayloadResults(string) = [
  Ok({data: "hi"}),
  Ok({data: "bye"}),
  Error("¡Algo incorrecto ha pasado!")
];

Tipos mutuamente recursivos

Igual que las funciones, los tipos pueden ser mutuamente recursivos usando and:

type student = {taughtBy: teacher}
and teacher = {students: list(student)};

Nota que no haya punto y coma al finalizar la primera linea y que no haya type en la segunda linea.

Decisiones de Diseño

Un sistema de tipos que permite argumentos de tipos esta básicamente permitiendo funciones a nivel de tipos. list(int) es en realidad la función de tipo list recibiendo el tipo int, y devolviendo el tipo concreto final, que usarás en algunos sitios. Abras notado que en otros lenguajes esto se llama mas o menos "genéricos". Por ejemplo, ArrayList<Integer> en Java.

La Regla del menos potente (The principle of least power) se aplica cuanto estás intentando "Acabar las cosas". Si el ámbito del problema lo permite, definitivamente escoge la solución disponible menos abstracta (ó más concreta), de esta manera la solución se alcanza mas rápido y tiene menos in-direcciones inestables que tendrás que recorrer. Por ejemplo, escoge tipos en vez de datos con forma flexible, escoge una configuración orientada a datos en vez de llamadas a funciones Turing completas, elige llamadas a funciones sobre macros, escoge macros en vez de copias de bifurcación (fork) de proyectos, etc. Cuando restringes tu ámbito y potencia, todo es más fácil de analizar. Esto es así, si el ámbito se restringe lo suficiente para permitirlo.

When a type system is an all-encompassing aspect of your program, we need to make sure we leave enough power in order not to overly constrain your expressiveness; without "type functions", you'd end up with quite a bit of boilerplate, e.g. hard-coded listOfInt, listOfString, listOfArrayOfFloat, their respective helper functions, etc. Sin embargo, por favor asegúrate que no abusas de la potencia que se te ofrece a través del potente sistema de tipos. Algunas veces, esta bien usar un poquito de plantilla para reducir la necesidad de la potencia extra de los tipos. ¡En todo caso, soluciones elegantes podrían mostrar tu pragmatismo y tu juicio mejor que tipos sofisticados!
