JSX

Te gustaría tener sintaxis de HTML en tu código de Reason? Si no, puedes saltar esta sección y pretender que no has visto nada!

Reason tiene soporte para la sintaxis de JSX con algunas pequeñas diferencias con la especificada en ReactJS. El JSX de Reason no esta ligado a ReactJS, se convierte en llamadas de función normales:

Nota para lectores que usen ReasonReact: esto no es en lo que ReasonReact convierte JSX al final. Ver sección de Uso para mas información.

Tag con Mayúscula

<MiComponente foo={bar} />

se convierte en

([@JSX] MiComponente.make(~foo=bar, ~children=[], ()));

Tag con Minúscula

<div foo={bar}> child1 child2 </div>;

se convierte en

([@JSX] div(~foo=bar, ~children=[child1, child2], ()));

Children

<MiComponente> foo bar </MiComponente>

Este es el sintaxis para pasar una lista de dos elementos, foo y bar, a la posicion de hijos. Se convierte en una lista conteniendo foo y bar:

([@JSX] MiComponente.createElement(~children=[foo, bar], ()));

Nota una vez más que este no es la transformación que ocurre para ReasonReact; ReasonReact convierte la lista final en un array pero la idea aún aplica.

Entonces naturalmente <MyComponent> foo </MyComponent> se convierte en ([@JSX] MyComponent.createElement(~children=[foo], ())). Hagas lo que hagas los argumentos en la posicion de hijos seran convertidos en lista. Y qué si no queremos eso? Que pasa si queremos pasar foo directamente sin esa transformación?

Children Spread

Para resolver el problema anterior, hemos introducido

<MiComponente> ...foo </MiComponente>

Esto pasa el valor foo sin convertirlo en una lista (o array, en el caso de ReasonReact). Esto se convierte en:

([@JSX] MiComponente.createElement(~children=foo, ()));

Esto es especialmente útil en los casos donde tienes un foo que es ya una lista de cosas y quieres pasarlo sin envolverlo en otra lista (lo cual sería un error de tipo) \ *. También nos permite pasar cualquier estructura de datos a la posición de children (Recuerda, JSX children es realmente sólo un prop normal):

<MiComponente> ...((theClassName) => <div className=elClassName />) </MiComponente>;
<MiForm> ...("Hello", "Submit") </MiForm>;

Uso

Vease ReasonReact para una aplicación de ejemplo de JSX, que transforma las llamadas de arriba en una llamada de ReasonReact.

Aquí está una etiqueta JSX que muestra la mayoría de las características.

<MyComponent
  booleanAttribute={true}
  stringAttribute="string"
  intAttribute=1
  forcedOptional=?{Some("hello")}
  onClick={send(handleClick)}>
  <div> {ReasonReact.stringToElement("hello")} </div>
</MyComponent>

Diferencias con el JSX de JS

  • Los atributos y los elementos hijos no requieren {}, pero los mostramos de todas formas para que sea mas fácil aprender.
  • No hay soporte para la extensión (spread) de props en JSX: <Foo {...bar} />. Aunque, un poco relacionado, si tenemos spread de los elementos hijos como descrito anteriormente: <Foo> ...baz </Foo>.
  • Punning!

Punning

"Punning" se refiere a la forma abreviada de sintaxis para cuando una etiqueta y un valor son iguales. Por ejemplo, en JavaScript, en lugar de hacer return {name: name}, puede hacer return {name}.

El JSX de Reason si tiene soporte para la abreviación (punning). <input checked /> es solo una abreviación de <input checked=checked />. El formateador ayudará a formatearlo simple cuando sea posible. Esto es conveniente en los casos donde hay un montón de props que pasar:

<MiComponente isLoading text onClick />

Por lo tanto, un componente de Reason JSX puede incluir unos props adicionales antes de tener que recurrir a soluciones externas.

Nota que esto es diferente de ReactJS JSX, que no tiene estas abreviaciones. En ReactJS, <input checked /> se convierte en <input checked=true />, para ajustarse a la especificacion del DOM y para tener compatibilidad con versiones anteriores.

Consejos & Trucos

Para los autores de bibliotecas que quieran aprovechar el JSX: el atributo [@JSX] arriba es un gancho para posibles macros de ppx que encuentren funciones queriendo formatearse como JSX. Una vez descubierta la función, se puede transformar en cualquier otra expresión.

De este modo, todos se benefician del sintaxis de JSX sin tener que optar por usar una biblioteca especifica que lo use, por ejemplo, ReasonReact.

Las llamadas a JSX tienen soporta para las caracteristicas de argumentos marcados: opcionales, pasados explicitamente y opcionales y opcionales con default.

Decisiones de Diseño

La manera en que diseñamos este JSX se relaciona mucho con la manera que nos gustaría ver este idioma evolucionar. Consulte la sección "¿Cual es el punto?" en este blog.

La capacidad de macros en el idioma + la sintaxis JSX que es independiente de paquetes de terceros permite a cada libreria tener JSX sin molestia. De esta manera, agregamos similitud visual al idioma subyacente, OCaml, sin comprometer su semántica (como ejecuta). Una de las mayores metas de Reason es permitir que mas personas puedan aprovechar el bello idioma de OCaml sin tener que desperdiciar tiempo preciado en debates sobre formato y sintaxis.

* Te preguntarás por que nunca necesitaste tal expansión de elementos hijos (children spread) en ReactJS; ReactJS usa una lógica especial en tiempo de ejecución + transformaciones sintácticas especiales + detención y marcado de argumentos variádica para evitar muchos de estos casos (ver aquí). Dicho uso dinámico complica bastante la detección del sistema de tipos. Dado que controlamos toda la sintaxis y ReasonReact, decidimos introducir la expansión de elementos hijos para diferenciar entre el caso de envolver o no envolver, sin penalización en tiempo de compilación y ejecución!
