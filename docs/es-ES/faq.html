Preguntas Frecuentes

¿Debo aprender Reason o OCaml primero?

¡No hay necesidad de escoger! Reason y OCaml comparten exactamente la misma semántica (es decir, cómo se ejecuta el código). Sólo la sintaxis difiere. Ten Reason-tools a mano, así puedes convertir libremente entre las dos sintaxis. Un tutorial de Reason es un tutorial de OCaml, y viceversa. En el terminal, puedes tener estos alias:

# convierte código de ocaml en reason
alias mlre="pbpaste | refmt --parse ml --print re --interface false | pbcopy"
# convierte código de reason en ocaml
alias reml="pbpaste | refmt --parse re --print ml --interface false | pbcopy"

Leen tu código desde el portapapeles (macOS), lo convierten, y lo pegan de nuevo en el portapapeles! Intercambia pbpaste y pbcopy por las funciones de portapapeles de su sistema.

No estoy seguro de qué hacer con Reason

Nosotros compilamos a JS muy bien. Piensa en qué proyecto harías normalmente si fuera con JavaScript puro; ¡en su lugar trata de portar o escribir lo mismo usando Reason y BuckleScript! Nosotros recomendamos tratar de hacer, proyectos concretos para el usuario final (e.g. una pequeña utilidad de línea de comandos) en vez de proyectos a nivel de infraestructura (e.g., un generador de plantillas). Esta última categoría requiere conocimientos y comprensión de código Reason idiomático.

¿Cuál es la relación entre la Reason, BuckleScript y OCaml?

Ver aquí. Reason es una sintaxis para OCaml y soporta todas sus características. BuckleScript compila código OCaml/Reason a JavaScript.

¿De dónde vienen todas estas funciones print_endline, string_of_int?

Son de la biblioteca estándar, pre abiertas (opened) durante la compilación de tu archivo. Por esta razón están disponibles en el ámbito global.

¿Puedo tener una función para imprimir estructuras de datos arbitrarias?

Si estas compilando a JavaScript a través de BuckleScript, puedes utilizar console.log de JS a través de Js.log. Si estás compilando a código nativo, necesitarás algo como ppx_show. Una futura característica de OCaml (llamada modular implicit) resolverá esto directamente en la lenguaje.

¿Por qué hay un + para añadir enteros y +. para añadir enteros en coma flotante, etc?

Ver here.

¿Funciona la biblioteca ___ con Reason?

La mayoría de las bibliotecas JS deben funcionar fácilmente con Reason y BuckleScript. En la parte nativa, como Reason es solo una sintaxis de transformación: si, ellas también funcionan con Reason. Pero el flujo nativo de trabajo es actualmente un proceso en construcción y necesita ser pulido.

¿Cual es la historia del lado del servidor? ¿Debería compilar a nativo o JS y utilizar node.js?

Nosotros compilamos a nativo, pero el flujo de trabajo nativo es actualmente un trabajo en progreso. At this time, we recommend compiling to JS through BuckleScript and use the JS wrappers at reasonml-community or somewhere else.

¿Cual es la historia de BuckleScript asíncrono?

Primero, si no estas usando interfaces de alguna biblioteca que usa promesas, simplemente puede utilizar callbacks. Todo el mundo los entiende y el rendimiento es muy alto.

If you need to bind to a JS library that uses promises, or communicate with such library, you can use BS's Js.Promise. También hay potencial para tener algo de endulzamiento sintáctico (syntactic sugar) en el futuro. A largo plazo, nos gustaría implementar una especificación de promesas compatible desarrollada apropiadamente en OCaml/Reason, y así hacer las optimizaciones del compilador efectivas.

Para una solución más idiomática de OCaml: del lado nativo de OCaml, tenemos lwt y Async. No lo utilizamos para web ahora mismo, pero podríamos en el futuro.

¿Qué es la historia de prueba (unitaria)?

Algunas de las características del lenguaje OCaml (no sólo tipos) podrían diferir en la necesidad de pruebas unitarias hasta más tarde. In the meantime, for compilation to JS, we're working on Jest wrapper. También veremos como usar Jest para código nativo, si Jest está escrito en Reason en el futuro (sin plan concreto todavía). Ahora mismo OUnit es una liviana y buena librería escrita en OCaml nativo.

¿Para qué es el archivo .merlin en la raíz de mi proyecto?

Es el archivo de metadatos de Merlin, el * backend* compartido de integración con el editor para autocompletar, ir a definición, etc. Para el flujo de trabajo de JavaScript, el sistema de construcción bsb genera el fichero .merlin por ti; No necesitas añadirlo a tu versión de control y no tienes que modificarlo manualmente.

No veo ningún import o require en mi archivo; ¿Cómo funciona la resolución de módulos?

Reason/OCaml no requieren escribir ningún import; las referencias a los módulos en el archivo se buscan automáticamente en el proyecto. Específicamente, un módulo Hello pide al compilador que busque el archivo hello.re o hello.ml (y su correspondiente archivo de interfaz, hello.rei o hello.mli, si está disponible).

Un nombre de módulo es el nombre del archivo, en mayúsculas. Tiene que ser único por proyecto; Este abstrae el sistema de archivos y te permite mover archivos sin cambiar el código.

Es Some | None, contents, Array, List y todo esto especial? ¿De dónde vienen?

Son definiciones ordinarias de módulos, registros o variantes que vienen con la biblioteca estándar, convenientemente abiertos (opened) por defecto durante la compilación.

¿Qué significa un argumento con un guion bajo antepuesto (e.g. _ o _foo)?

Say you have List.map(item => 1, myList);. El argumento item no se utiliza y el compilador generará una advertencia. Using item => 1 instead indicates that you're intentionally receiving and ignoring the argument, therefore bypassing the warning. Alternatively, _item => 1 has the same effect, but indicates more descriptively what you're ignoring.

¿Qué es este MyModule.t que sigo viendo?

Asumiendo que MyModule es un nombre de módulo, t es una convención de la comunidad que indica "el tipo que representa a ese módulo, lo que sea que signifique". Por ejemplo, para el módulo Js.String, String.t es el tipo que se usara y que representa "una cadena".

¿Por qué hay un Js_promise y entonces un Js.Promise? Y acerca de Js_array, Js_string y cualquier otro?

Como convención, Js_foo es el módulo real, y Js.Foo es sólo un alias para el. Son equivalentes. Es preferible Js.Foo, por que es el nombre del módulo público oficial.

¿Cuándo estarán listos implícito modular, multinúcleo y efectos algebraicos?

Estarán un día. Mientras tanto, ¡ayúdanos a crear más código Reason! La popularidad ayudará a atraer más contribuciones de OCaml. Cuanto menos necesite preocuparse la gente de OCaml de las partes esenciales, ¡más podrán concentrarse en investigación importante y ejecución!

¿Por qué BuckleScript y bsb son tan rápidos? ¿Cómo puedo ralentizarlo?

BuckleScript está optimizado en rendimiento en todo el stack. Puedes intentar hacerlo mas lento añadiendo multiples capas de indirecciones y meta-programación. Intenta:

  • Añadiendo unos pocos bucles infinitos aquí y allá.
  • Rellenando con alguna herramienta de compilación de JavaScript el * pipeline*.
  • Introduciendo más dependencias para escribir un Hola mundo.

Estoy viendo un extraño archivo .cmi/.cmx/.cmj/.cma referenciado en un error del compilador. ¿De dónde vienen estos archivos?

La comunidad de OCaml con frecuencia utiliza extensiones de archivo para distinguir entre tipos de origen, artefactos y metadatos, dependiendo de lo que quieras crear (nativo/bytecode/JavaScript). El siguiente es un resumen de algunas de las extensiones de archivo que te puedes encontrar:

Archivos fuente

  • .ml: archivo de código fuente OCaml
  • .mli: archivo de interfaz de OCaml; determina qué partes del archivo de coincidencias .ml son visibles para el mundo exterior
  • .re: archivo de código fuente de Reason. Como .ml, pero para Reason
  • .rei: archivo de interfaz de Reason. Como .mli, pero para Reason

Archivos compilados

  • .cmi: archivo de interfaz compilado (.rei/mli)
  • .cmx: archivo de objetos compilado para salida nativa (vía ocamlopt)
  • .cmo: archivo de objetos compilado para salida bytecode
  • .cmj: archivo de objetos compilado para web (vía BuckleScript)
  • .cma: archivo de biblioteca para salida bytecode (equivalente a archivos .a de C)
  • .cmax: archivo de biblioteca para salida nativa
  • .cmt: contiene un "Typedtree" – básicamente el AST con toda información de tipos
  • .cmti: igual que un archivo .cmt, pero para archivos de interfaz
  • .cmxs: plugin cargado dinámicamente (para compilación nativa)
  • .o: archivo de objetos nativo compilado
  • .out: nombre/extensión convencional para la salida final producido por ocamlc/ocamlopt (e.g. ocamlc -o myExecutable.out)

Otros archivos del ecosistema OCaml

  • .mll: archivo de definición del analizador léxico ocamllex
  • .mly: archivo de definición del generador de analisis de ocamlyacc
  • .mldylib: contiene una lista de rutas de módulo que serán compilados y archivados juntos para construir el correspondiente .cmxs requerido (plugin nativo)
  • .mliv: ficheros de Batteries-specific para algunos preprocesados personalizados.
  • .mllib: biblioteca Ocaml (cma and cmxa)
  • .mlpack: paquete Ocaml (cmo construido con el modificador the -pack)
  • .mlpp: ficheros Extlib-specificos para algunos preprocesados personalizados
  • .mltop: fichero *top-level * OCamlbuild, usado por OCamlbuild para generar un fichero .top
  • .odocl: fichero documentación OCaml

Si algunas de estas explicaciones todavía son un poco crípticas, aquí hay extensiones de algunos de los términos usados anteriormente: - AST: Arbol de sintaxis abstracta. La estructura de datos del código fuente, en la que opera el compilador. - Linking : la etapa en la que el compilador toma muchos archivos compilados intermedios y los ensambla juntos. Por ejemplo, unir A con B, porque el fichero fuente de origen A se refirió a B. -Nativo: Builds que corren instrucciones ensamblador esenciales de la plataforma en cuestión. - Bytecode: como código nativo, pero más portable y menos eficiente - Object file: contiene código máquina que no es directamente ejecutable.

Hay más información y contexto para muchas de estas extensiones de archivos en el sitio de OCaml y en esta entrada de lista de correos. También hay nociones más profundas sobre compilación nativa y compilación bytecode que contiene descripciones más detalladas en el manual de OCaml.
