Module

基本

モジュールは小さなファイルのようです！それらは型定義、let 束縛、ネストされたモジュールなどを含めることができます。

作成

モジュールを作成するには、module キーワードを使います。 モジュールの名前は大文字で始める必要があります。 .re ファイルに置けるものは何でも、モジュール定義の {} ブロックの内側に置けます。

module School = {
  type profession = Teacher | Director;

  let person1 = Teacher;
  let getProfession = (person) =>
    switch (person) {
    | Teacher => "A teacher"
    | Director => "A director"
    };
};

モジュールの内容（型を含む！）には、レコードのように、ドット（.）記法を使ってアクセスできます。これは名前空間のためのモジュールのユーティリティの説明です。

let anotherPerson: School.profession = School.Teacher;
print_endline(School.getProfession(anotherPerson)); /* "A teacher" */

ネストされたモジュールも機能します。

module MyModule = {
  module NestedModule = {
    let message = "hello";
  };
};

let message = MyModule.NestedModule.message;

モジュールを開く

モジュールの値/型へ毎度毎度参照することは面倒になることがあります。モジュールの定義を開き、モジュール名の記載を省略して、それ自身の内容を参照することができます。ふたつの方法があります。

ローカルに開く。

let message =
  School.(
    switch (person1) {
    | Teacher => "Hello teacher!"
    | Director => "Hello director!"
    }
  );

グローバルに開く。それは（訳注: それがどのモジュールのものであるかを）容易に推測できるところで役立つ時に、必要最小限で使ってください。

open School;
let anotherPerson: profession = Teacher;
printProfession(anotherPerson);

モジュールの拡張

モジュールで include 使うと、静的にモジュールの内容を新しいものとして「スプレッド」します。結果的に、しばしば「継承」や「ミックスイン」の役割を果たします。

module BaseComponent = {
  let defaultGreeting = "Hello";
  let getAudience = (~excited) => excited ? "world!" : "world";
};

module ActualComponent = {
  /* 内容がコピーされます */
  include BaseComponent;
  /* BaseComponent.defaultGreeting をオーバーライドします */
  let defaultGreeting = "Hey";
  let render = () => defaultGreeting ++ " " ++ getAudience(~excited=true);
};

注意：open と include は、非常に異なるものです！ 前者はモジュールの内容をあなたの現在のスコープに持ってくるので、毎回モジュール名を前置して値を参照する必要がありません。 後者はモジュールの定義を静的にコピーしてから、その後 open します。

すべての .re ファイルはモジュール

OCaml/Reason ファイルはモジュールにマップされます。これは、以前、他の言語ではコード生成を必要としたいくつかの面白い表現性を解き放ちます。 React.re というファイルは、暗黙的に他のソースファイルが認識できるモジュール React を形成します。

/* FileA.re. これは通常、下記のモジュール FileA にコンパイルされます。 */
let a = 1;
let b = 2;

/* FileB.re */
/* FileA の実装を新しい API にマップします。 */
let alpha = FileA.a;
let beta = FileA.b;

この企まれたスニペットは、ファイルの「コピー」を表現したものです。

/* FileA.re. これは通常、下記のモジュール FileA にコンパイルされます */
let a = 1;
let b = 2;

/* FileB.re */
/* ランタイムのオーバーヘッドなしに、fileA.re の内容に正確にコンパイルします！ */
include FileA;

注意：ファイルはモジュールなので、ファイル名は、慣習的には、モジュール名と合わせた大文字にした方が良いです。 大文字でないファイル名は不正ではありませんが、大文字のファイル名に変換されます。 つまり、file.re はモジュール File にコンパイルされることになります。 ここでの食い違いを単純化かつ最小化するために、慣習でもファイル名を大文字にすることにしています。

シグネチャ

モジュールの型はシグネチャと呼ばれ、明示的に記述することもできます。 もしモジュールが .re（実装）ファイルのようなものならば、モジュールのシグネチャは、.rei（インターフェイス）ファイルのようなものです。

作成

シグネチャを作るには、module type キーワードを必要とする使用します。 シグネチャの名前は大文字で始める必要があります。 .rei ファイルに置けるものは何でも、シグネチャ定義の {} ブロックの内側に置けます。

/* 前のセクションの例を引っ張ってきます */
module type EstablishmentType = {
  type profession;
  let getProfession: profession => string;
};

シグネチャは、シグネチャと一致するモジュールであるためにモジュールが満たすべき要求のリストを定義します。それらの要求は次の形式です。

  • let x: int; は、int 型の x と名付けられた let 束縛を要求します。
  • type t = someType; は、someType と等しくなる t 型フィールドを要求します。
  • type t; は、t 型フィールドを要求しますが、実際の t の具体的な型の要求はありません。 let makePair: t => (t, t) のように、シグネチャ内の別の箇所で、関係を説明するために t を使うと思いますが、例えば t が int であると仮定することはできません。 これにより、素晴らしい、強制力のある抽象化能力がもたらされます。

様々な種類の型エントリを説明するために、モジュールを要求する上記のシグネチャ EstablishmentType について考えます。

  • profession と名前の付いた型を宣言します。
  • profession 型の値を受け取り、文字列を返す関数を含めなければなりません。

注意：

EstablishmentType 型のモジュールには、前のセクションに出てきた School モジュールのように、シグネチャ宣言よりも多くのフィールドを含むことができます。（もしそれに EstablishmentType 型を割り当てることを選ぶなら。 さもなければ、School は、すべてのフィールドを公開します。） これは、効果的に、実装の詳細が強制された person1 フィールドを作ります。 それはシグネチャに存在しないため、部外者はそれにアクセスすることはできません。シグネチャは他人がアクセスできるものを 制限しました 。

EstablishmentType.profession 型は、抽象型です。具体的な型を持ちません。「実際の型は気にしませんが、getProfession の入力として使用されます」と言っています。 これは同じインターフェイスの下で多くのモジュールに適合させるのに便利です。

module Company: EstablishmentType = {
  type profession = CEO | Designer | Engineer | ...;

  let getProfession = (person) => ...;
  let person1 = ...;
  let person2 = ...;
};

それは、他に頼ることができない実装の詳細に基づいた型を隠すのにも便利です。 もしあなたが Company.profession の型について尋ねる場合、ヴァリアントを公開する代わりに、それは「Company.professionです」とだけ教えるでしょう。

モジュールシグネチャの拡張

モジュールそれ自身のように、モジュールシグネチャでも include module type of Foo をによって拡張できます。

module type BaseComponent = {
  let defaultGreeting: string;
  let getAudience: (~excited: bool) => string;
};

module type ActualComponent = {
  /* シグネチャがコピーされます */
  include (module type of BaseComponent);
  let render: unit => string;
};

すべての .rei ファイルはシグネチャ

React.re ファイルが、暗黙的に React モジュールを定義するのと同様に、React.rei ファイルは、暗黙的に React のためのシグネチャを定義します。 React.rei が提供されていない場合、React.re のシグネチャは、デフォルトでモジュールのすべてのフィールドを公開します。 それらは実装ファイルに含めていないので、.rei ファイルは、それらの該当するモジュールのパブリック API を文書化するエコシステムで使用されます。

/* React.re ファイル（実装。React モジュールにコンパイルされる。） */
type state = int;
let render = (str) => str;

/* React.rei ファイル（インターフェイス。React モジュールのシグネチャにコンパイルされる） */
type state = int;
let render: str => str;

モジュール関数（ファンクタ）

モジュールは関数に渡すこともできます！ それは第一級のものとしてファイルを渡すことと同等になるでしょう。 ただし、モジュールは、他の一般的なコンセプトとは異なる言語の「レイヤ」にいます。つまり、正規の関数にそれらを渡すことはできません。 代わりに、「ファンクタ（functors）」と呼ばれる特別な関数にそれらを渡します。

ファンクタを定義し、利用するためのシンタックスは、正規の関数を定義し、利用するためのシンタックスに非常に似ています。主な違いは次のとおりです。

  • ファンクタは、let の代わりに module キーワードを使います。
  • ファンクタは、引数としてモジュールを受け取り、モジュールを返します。
  • ファンクタは、注釈付きの引数を要求します。
  • ファンクタは、大文字から始めなければなりません（ちょうどモジュール/シグネチャのように）。

これは、Comparable 型のモジュールを受け取り、比較可能なアイテムのようなものを含むことができる新しいセットを返す MakeSet ファンクタの例です。

module type Comparable = {
  type t;
  let equal: (t, t) => bool;
};

module MakeSet = (Item: Comparable) => {
  /* 私たちのネイティブなバッキングデータ構造として、リストを使いましょう。 */
  type backingType = list(Item.t);
  let empty = [];
  let add = (currentSet: backingType, newItem: Item.t) : backingType =>
    /* もしアイテムが存在していれば */
    if (List.exists((x) => Item.equal(x, newItem), currentSet)) {
      currentSet /* 同じ（イミュータブルな）セット（本当にリスト）を返します。 */
    } else {
      [
        newItem,
        ...currentSet /* セットに prepend して、返します。 */
      ]
    };
};

ファンクタは関数適用のシンタックスを使っても適用できます。このケースでは、項目が整数のペアであるセットを作っています。

module IntPair = {
  type t = (int, int);
  let equal = ((x1, y1), (x2, y2)) => x1 == x2 && y1 == y2;
  let create = (x, y) => (x, y);
};

/* IntPair は、MakeSet で要求された Comparable シグネチャに従います。 */
module SetOfIntPairs = MakeSet(IntPair);

モジュール関数の型

モジュールの型と同様、ファンクタの型も、ファンクタだと想定しているものを制約したり、隠したりするように作用します。 ファンクタの型のためのシンタックスは、関数の型のためのものと整合性が取れていますが、ファンクタが引数としてモジュールを受け入れ、値を返すシグネチャを表現するために、型は大文字にされます。 先の例では、セットのバッキング型を公開しています。MakeSet にファンクタのシグネチャを与えることで、基となるデータ構造を隠すことができます！

module type Comparable = ...

module type MakeSetType = (Item: Comparable) => {
  type backingType;
  let empty: backingType;
  let add: (backingType, Item.t) => backingType;
};

module MakeSet: MakeSetType = (Item: Comparable) => {
  ...
};

欠点

モジュールとファンクタは、残りの部分（関数、let 束縛、データ構造など）とは異なる言語の「レイヤ」にいます。 例えば、それらをタプルやレコードに簡単に渡すことはできません。 賢明なご利用を！ 多くの場合、ただのレコードや関数で事足ります。

module School = {...};

/* これは真っ当にシンタックスエラーになります！ */
let schools = (School, School);
