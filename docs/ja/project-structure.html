Project Structure

コミュニティには明文化されていない既存の慣習があります (訳者注: OCaml コミュニティの慣習だと思われる) 。 その中には従わなくてよい慣習もあり、従うべき慣習もあります。 早速、簡単に実行できる慣習を紹介しましょう。

ファイル名

ファイル名はキャメルケースにします (先頭が大文字のアッパーケース) 。

理由: モジュール名は必ず大文字で始まります。 初心者はファイルとモジュールの関係についてたびたび質問します。ファイルはどうモジュールに結びつけられるのか、 draw.re という名前のファイルがなぜ Draw モジュールとして扱われるのか。先頭を小文字にした識別子でモジュールを参照しようとします。 ファイル名を Draw.re にしましょう。これならファイル名とモジュール名が同じになります。 uRI.re のような奇妙な名前のファイルを避けられます (訳者注: OCaml では URL という名前のモジュールを作りたいなら、ファイル名を uRI.ml にする必要がある) 。

.merlin ファイルを無視する

.merlin ファイルはビルドシステムによって生成されるため (少なくとも JavaScript のワークフローでは) 、手動で編集すべきではありません。リポジトリに追加しないでください。

理由: .merlin は Merlin (訳者注: OCaml 用のエディタ支援ツール) が使用するファイルで、プロジェクトのレイアウトが記述されます。 このファイルに記述されるパスはクロスプラットフォームではない絶対パスです (例えば Windows のパスは他の OS と異なります) 。

ディレクトリ

あまりに深い階層構造は避けましょう。できるだけフラットな構造を保ち、ファイル数を抑えましょう (注: モジュールはネスト可能です) 。

理由: ディレクトリ構造は大きな悩みの種になり得ます。 「階層に意味を持たせるにはどうすればいいでしょうか」と私たちは何度も尋ねられます。 自転車置き場の議論のコストは高くつきます。

サードパーティーへの依存

最小限に留めましょう。

理由: コンパイルを行う静的型付け言語は、動的言語のように軽々しく複雑な依存関係を持てません。特に今はまだ (中期的にかかる学習コストを減らすために) NPM/Yarn を利用しており、これらのツールは Reason や BuckleScript で開発されているわけではありません。 依存関係をシンプルかつ小さく保てば、衝突の可能性が減ります (ダイヤモンド依存関係やインターフェースの矛盾など) 。

サードパーティーのコードは dependencies に記述しましょう。 peerDependencies に記述するのは避けましょう。

理由: 一日を費やした実装と考えて丁重に扱うべきです。

ドキュメント

用意しましょう。 Spend more effort making them great (examples, pitfalls) and professional rather than just fancy-looking. Do use examples, and avoid using names such as foo and bar. There's always more concrete names (it's an example, no need to be abstract/generalized just yet. The API docs will do this plentily). For blog posts, don't repeat the docs themselves, describe the transition from old to new, and why (e.g. "it was a component, now it's a function, because ...").

Justification: It's hard for newcomers to distinguish between a simple/decent library and one that's fancy-looking. For the sake of the community, don't try too hard to one-up each other's libraries. Do spread the words, but use your judgement too.

PPX やその他のメタツール

最小限の使用に留めましょう。 PPX, unless used in renown cases (printer, accessors and serializer/deserializer generation), can cause big learning churn for newcomers; on top of the syntax, semantics, types, build tool & FFI that they already have to learn, learning per-library custom transformations of the code is an extra step. More invasive macros makes the code itself less semantically meaningful too, since the essence would be hiding somewhere else.

パラダイム

Don't abuse overly fancy features. Do leave some breathing room for future APIs but don't over-architect things.

Justification: Simple code helps newcomers understand and potentially contribute to your code. Contributing is the best way for them to learn. The extra help you receive might also surpass the gain of using a slightly more clever language trick. But do try new language tricks in some of more casual projects! You might discover new ways of architecting code.

配布

If it's a wrapper for a JS library, don't publish the JS artifacts. 正当なライブラリ (訳者注: Reason や OCaml 用？) であれば、 JavaScript ユーザーが使うであろう "lib/js" に成果物を配布してください。 既存の実装を壊さないようにして JavaScript ライブラリを徐々に Reason + BuckleScript に変換する場合は特にそうすべきです。

Do put the keywords "reason" and "bucklescript" in your package.json keywords field. This allows us to find the library much more easily for future purposes.

Justification: Be nice to JS consumers of your library. They're your future Reasoners.
