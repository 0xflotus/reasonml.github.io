JSX

あなたの Reason コードに HTML のようなシンタックスはありませんか？もしないなら、すぐにこのセクションをスキップして、何も見なかったことにしてください！

Reason は ReactJS のものと比較して若干違いのある、JSX シンタックスをサポートします。 Reason JSX は ReactJS に紐付いてはいません。Reason JSX は通常の関数呼び出しに翻訳します。

ReasonReact の読者のための注意。これは ReasonReact が最終的に JSX に変えるものではありません。使い方セクションで詳細を参照してください。

大文字のタグ

<MyComponent foo={bar} />

以下になります。

([@JSX] MyComponent.createElement(~foo=bar, ~children=[], ()));

小文字のタグ

<div foo={bar}> child1 child2 </div>;

以下になります。

([@JSX] div(~foo=bar, ~children=[child1, child2], ()));

Children

<MyComponent> foo bar </MyComponent>

これは foo と bar の2要素を子に持つリストを渡すシンタックスです。foo と bar を含んでいるリストにデシュガーします。

([@JSX] MyComponent.createElement(~children=[foo, bar], ()));

これは ReasonReact のためのトランスフォームではないことに再注意してください。ReasonReact は最後のリストを配列に変更します。しかし、このアイディアはまだ適用されます。

したがって当然、<MyComponent> foo </MyComponent> は ([@JSX] MyComponent.createElement(~children=[foo], ())) にデシュガーします。 すなわち、あなたが何をやっても、子の位置に渡される引数はリストにラップされます。 あなたがそれを望まない場合は？ もし、直接 foo を余分なラッピングをせずに渡したい場合は？

子のスプレッド

上記の問題の解決する方法についての紹介です。

<MyComponent> ...foo </MyComponent>

これは foo という値をリスト（または ReasonReact では配列）にラップせずに渡します。これは下記のようにデシュガーします。

([@JSX] MyComponent.createElement(~children=foo, ()));

これは、すでにあるもののリストである foo を処理して、余計な時間（型エラーになる）をかけずにそれを届けたいケースで有用です。 任意のデータ構造を children の位置（JSX の children は本当にただの prop であることを覚えておいてください）に渡すこともできます。

<MyComponent> ...((theClassName) => <div className=theClassName />) </MyComponent>;
<MyForm> ...("Hello", "Submit") </MyForm>;

使い方

JSX のアプリケーションの例については ReasonReact を参照してください。上記の呼び出しを ReasonReact 固有の呼び出しに変換します。

ほとんどの機能を示す JSX タグです。

<MyComponent
  booleanAttribute={true}
  stringAttribute="string"
  intAttribute=1
  forcedOptional=?{Some("hello")}
  onClick={send(handleClick)}>
  <div> {ReasonReact.stringToElement("hello")} </div>
</MyComponent>

JS JSX の先へ

  • 属性と子は {} を必須とはしませんが、学習を容易にするためにとりあえず表記します。一度、あなたのファイルを refmt すると、それらのいくつかは消え、いくつかは括弧に変わります。
  • <Foo {...bar} /> のような JSX prop のスプレッドはサポートされていません。が、やや関連があります。上記で説明した <Foo> ...baz </Foo> のような子のスプレッドはサポートしています。
  • Punning!

Punning

「Punning」はラベルと値が同じ場合のシンタックスの省略系です。 例えば、JavaScript で言うと、return {name: name} と書く代わりに、return {name} と書けます。

Reason JSX は punning をサポートします。 <input checked /> は、<input checked=checked /> の単なる省略形です。 フォーマッタは可能な限り後者にフォーマットすることを支援するでしょう。 これは多くの props を渡す時に便利です。

<MyComponent isLoading text onClick />

その結果、Reason JSX コンポーネントでは、props を渡すのを避けるための別のライブラリを採用する前に、もう少し props を詰め込むことができます。

これは ReactJS JSX の先へであって、ReactJS JSX には punning がないことに注意してください。 ReactJS の <input checked /> は、DOM のイディオムと後方互換性に準拠する目的で <input checked=true /> にデシュガーします。

ヒントとコツ

JSX を活用したいと思っているライブラリ作者へ。上記の [@JSX] 属性は、潜在的な ppx マクロが JSX としてフォーマットしたい関数を特定するためのフックです。 一度関数を特定すれば、それを他の式にできます。

これにより、それを使えば特定のライブラリ（例えば ReasonReact）を選択せずとも、誰もが JSX シンタックスの恩恵を受けることができます。

JSX 呼び出しはラベル付けされた関数の機能をサポートします。明示的に渡されるオプションとデフォルト値を持つオプション。

設計方針

私たちがこの JSX で設計した方針は、私たちがどのように言語の進化を支援したいかに関わってます。 このブログ記事 の「What's the point?」を参照してください。

言語にマクロがある能力 + ライブラリに依存しない JSX シンタックスは、すべてのライブラリが手間をかけることなく JSX を潜在的に保有することを可能にします。 このようにして、私たちは、それ自身のセマンティクス（または実行方法）を損なうことなく、基となる OCaml 言語にいくつかの視覚的な馴染みやすさを追加します。 Reason の大きな目標のひとつは、シンタックスやフォーマッティングに関する時間の無駄な議論を破棄するとともに、より多くの人々が美しい言語、すなわち OCaml の有利性を利用できるようになることです。

ReactJS では子のスプレッドのようなことが必要だったことがなかった理由について不思議に思うかもしれません。ReactJS はこれらのケースのほとんどを避けるために、いくつかの特殊なランタイムロジック + 特殊なシンタックス変換 + 可変引数の検出 & マーキングを使用しています（ここを参照してください）。 このような動的な使い方は型システムの検出をかなり複雑にします。 私たちは構文全体と ReasonReact を制御するため、コンパイル時間とランタイムのコストを除いて、ラッピングするケースとラッピングしないケースとの曖昧さをなくすために、子のスプレッドを紹介することを決めました！
