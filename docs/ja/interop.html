Interop

あなたの Reason コードの途中で、JavaScript にダンプするだけです。

If you're just hacking things together, this can be very nice, but you also have all of the unsafety of JavaScript code

Js.log("this is reason");

[%%bs.raw {|
console.log('here is some javascript for you');
|}];

{| と |} は OCaml の複数行文字列のデリミタです。 そこにタグを置くこともできます。例えば {something|、次に、閉じるためにマッチする |something} を探します。

そして、結果として得られる javascript が下記です。

// Generated by BUCKLESCRIPT VERSION 1.7.4, PLEASE EDIT WITH CARE
'use strict';
console.log("this is reason");
console.log('here is some javascript for you');

いくつかの JavaScript をダンプし、Reason からアクセスできるようにします。

Reason コードから使うことができる値が欲しい場合はどうしますか？

Js.log("this is reason");
let x = [%bs.raw {| 'here is a string from javascript' |}];
Js.log(x ++ " back in reason land"); /* ++ は文字列結合のための演算子です */

今、次のように疑問に思っているかもしれません。「これはどんな魔法？？ Ocaml はどのようにして x が文字であるとわかったの？」そうではありません。 このコードの x の型は何かと統合される魔法の型です！ これは非常に危険で、OCaml の型推論アルゴリズムに連鎖的な影響をもたらす可能性があります。

let y = [%bs.raw {| 'something' |}];
Js.log(("a string" ++ y, 10 + y));
/* 危険！！ ocaml は2つの全く異なる型として y を使用しても、あなたを止めてくれません */

これを修正するには、bs.raw の結果に対して、常に具体的な型を示す必要があります。

let x: string = [%bs.raw {| 'well-typed' |}];
Js.log(x ++ " back in reason land");
/* ocaml は x を文字列とは異なる別の型として x を使用しようとしたら、エラーを送出します */

そして、これが出力結果です！

// Generated by BUCKLESCRIPT VERSION 1.7.4, PLEASE EDIT WITH CARE
'use strict';
console.log("this is reason");
var x = ( 'here is a string from javascript' );
console.log(x + " back in reason land");
var y = ( 'something' );
console.log(/* tuple */[
      "a string" + y,
      10 + y | 0
    ]);
var x$1 = ( 'well-typed' );
console.log(x$1 + " back in reason land");

前のセクションの2つの %% と、ここの1つの % の違いは重要です！ [%%something ...] は、トップレベルの文を表す OCaml の「拡張ポイント」です（それは例えば、関数の中や値には現れません）。 [%something ...] は式の代わりをする拡張ポイントで、どこにでも置かれる可能性があります -- が、中に置いた JavaScript が実際に式であるかを確認してください！ 例えば、セミコロンをその後に置いたり、JavaScript の実行結果を得た時にシンタックスエラーが発生したりしないようにしてください。

関数をダンプする & 値を渡す

Bucklescript の様々な値のランタイム表現を機能させるために少し知識を必要とします。

  • strings は文字列、ints と floats は単に数値です。
  • 配列 は OCaml ではミュータブルな固定長のリストで、プレーンな javascript の配列として表現されます。
  • リスト はイミュータブルな関数型スタイルの連結リストで、間違いなくほとんどのケースで使用する慣用的なものです。 However, its representation is more complicated (try Js.log([1,2,3,4]) to check it out). このため、javascript とやり取りする時は大抵、 Array.of_list を介して Array へ、または Array.to_list<code> を介して <code>Array から変換します。
  • より深めたい場合は、BuckleScript のドキュメントにリストがあります。

配列を受け取り、数値を返す関数を、全く苦労することなく JavaScript で書けることを知っています。

let jsCalculate: (array(int), int) => int = [%bs.raw
  {|
 function (numbers, scaleFactor) {
   var result = 0;
   numbers.forEach(number => {
     result += number;
   });
   return result * scaleFactor;
 }
|}
];

let calculate = (numbers, scaleFactor) => jsCalculate(Array.of_list(numbers), scaleFactor);

Js.log(calculate([1, 2, 3], 10)); /* -> 60 */

もちろん、Reason にJavaScript で書いたこの関数は、手間なく Reason に移植することができました。

これは学習の際にとても有用な非常口であることを忘れないでください。すぐに飛びついて、何かを作ることができますが、立ち戻って、コンバートしたものを素晴らしい型安全な Reason コードに戻すのは良いエクササイズになります。

I've run into more than a few bugs because of raw JavaScript that I added to save time

落ち着いて物事を学ぶ

これまで、私達は非常に速く、緩い方法で、プロダクションには適さない、bs.raw を使用してきました。

しかし、実際に JavaScript にある関数を呼び出す必要がする場合、どうすれば良いでしょう？ それは DOM や node モジュールの使用等で相互作用するために必要になります。 BuckleScript では、external 宣言を使用します（ドキュメント）。

値を取得したり、関数を取得したりすることの両方とも、とても簡単です。

[@bs.val] external pi : float = "Math.PI";
let tau = pi *. 2.0;
[@bs.val] external alert : string => unit = "alert";
alert("hello");

しかし、何かもっと複雑な何かが欲しい時はどうでしょうか？Canvas の DOM ノードで getContext を呼び出す方法は以下のとおりです。

type canvas;

type context;

/* これらの抽象化した型を残しているのは、
 * どこにおいても直接使用することはないためです */
[@bs.send] external getContext : (canvas, string) => context = "";

let myCanvas: canvas = [%bs.raw {| document.getElementById("mycanvas") |}];

let ctx = getContext(myCanvas, "2d");

さて、何が起こっているか荷解きしましょう。Canvas DOM ノードと、関連付けられた RenderingContext オブジェクトのために、いくつかの抽象型が作成されました。

そして、getContext 関数を作成しましたが、@bs.val の代わりに、@bs.send を使用し、external のテキストに空文字列を使用しました。 @bs.send は「私達は第一引数上のメソッドを呼び出しています」ということを意味します。このケースでは（第一引数は） canvas です。 上記を踏まえて、BuckleScript は getContext(theFirstArgument, theSecondArgument) を heFirstArgument.getContext(theSecondArgument, ...) に翻訳します。

空文字列は「JS 上の名前は BuckleScript 上で external に与えた名前と同じです」ということを意味します。 – このケースでは（external に与えた名前は） getContext です。 他の名前（getRenderingContext のような）を付けたい場合は、BuckleScript が正しい関数を呼べるように、"getContext" という文字列を指定する必要があります。

面白い関数をもうひとつ追加しましょう。

[@bs.send] external fillRect : (context, float, float, float, float) => unit = "";

そして今、何かを描くことができるようになりました！

fillRect(ctx, 0.0, 0.0, 100.0, 100.0);

それほど多くはありませんが、他の canvas メソッドを追加することは似ており、いくつかの本当に楽しいことをやり始めることができます。

コンパイルされた JavaScript の見た目はどうでしょうか？

'use strict';
var tau = Math.PI * 2.0;
alert("hello");
var myCanvas = ( document.getElementById("mycanvas") );
var ctx = myCanvas.getContext("2d");
ctx.fillRect(0.0, 0.0, 100.0, 100.0);

おぉ！BuckleScript が pi 変数を私達のためにどのようにインライン化したか注目してもらえますか？出力結果も手動で書かれたものとほぼ同じように見えます。

既存の JavaScript ライブラリを使用する

人々が特定の JavaScript ラッパーを書く時、彼らは大抵 npm に公開すると思います。Libraries に進み、それらを見つける方法を探してください。

既存のラッパーのないライブラリを使う場合でも、いつもどおり最初に npm パッケージをインストールする必要があります。例えば、npm install --save <package-name> を使用し、それから、ラッパーを書いてください。 おそらく、特に便利な bs.module のFFI 機能を見つけることになると思います。それは JS コンパイルのターゲットのモジュール形式に応じて、適切な import や require を発火します。

一例として、ここに（Reason にコンバートされた、オリジナルが OCaml の）bs.glob ラッパーのソースコード全体を記載します。

type error;

[@bs.module] external glob : (string, (Js.nullable(error), array(string)) => unit) => unit = "";

[@bs.val] [@bs.module "glob"] external sync : string => array(string) = "";

そして、package.json の関連部分です。

{
  "name": "bs-glob",
  "version": "0.1.0",
  ...
  "devDependencies": {
    "bs-platform": "^1.9.1"
  },
  "dependencies": {
    "glob": "^7.1.2"
  }
}
