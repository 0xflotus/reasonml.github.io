JSX

Voudriez-vous un peu de syntaxe HTML dans votre Reason ? Si non, sautez vite cette section et faites semblant de n'avoir rien vu !

Reason supporte la syntaxe JSX, avec quelques légères différences par rapport à celle utilisée dans ReactJS. Le JSX de Reason n'est pas lié à ReactJS ; il se traduit en classiques appels de fonctions :

Note pour les lecteurs de ReasonReact : ce n'est pas la manière dont ReasonReact transforme le code JSX, au final. Voir la section Utilisation pour plus d'informations.

Balise avec une majuscule

<MyComponent foo={bar} />

devient

([@JSX] MyComponent.make(~foo=bar, ~children=[], ()));

Balise sans majuscule

<div foo={bar}> child1 child2 </div>;

devient

([@JSX] div(~foo=bar, ~children=[child1, child2], ()));

Enfants

<MyComponent> foo bar </MyComponent>

C'est la syntaxe pour passer une liste de deux éléments, foo et bar, comme enfants. Cela « désucre » une liste contenant foo et bar :

([@JSX] MyComponent.createElement(~children=[foo, bar], ()));

Notez à nouveau que ce n'est pas la transformation appliquée par ReasonReact. ReasonReact transforme la liste finale en tableau. Mais l'idée reste la même.

Donc naturellement, <MyComponent> foo </MyComponent> devient ([@JSX] MyComponent.createElement(~children=[foo], ())). Autrement dit, quoi que vous fassiez, les paramètres passés comme enfants seront placés dans une liste. Et si vous ne le souhaitez pas ? Qu’en est-il si vous voulez passer directement foo sans l’englober dans quoi que soit ?

Spread d'enfants

Pour répondre cette problématique, nous avons introduit :

<MyComponent> ...foo </MyComponent>

Ceci passe la valeur foo sans l’englober dans une liste (ni dans un tableau, dans le cas de ReasonReact). Dit autrement dit, cela devient :

([@JSX] MyComponent.createElement(~children=foo, ()));

C’est extrêmement utile dans le cas où vous avez sous la main foo qui est déjà une liste, et vous voulez la passer sans l’inclure dans une nouvelle liste (ce qui constituerait une erreur de type) *. Cela vous permet également de passer des structures de données arbitraires comme children (pour rappel, en JSX children n’est qu’une simple propriété) :

<MyComponent> ...((theClassName) => <div className=theClassName />) </MyComponent>;
<MyForm> ...("Hello", "Submit") </MyForm>;

Utilisation

Voir ReasonReact pour un exemple d’application de JSX, qui transforme les appels ci-dessus en un appel spécifique à ReasonReact.

Voici une balise JSX qui montre la plupart des fonctionnalités.

<MyComponent
  booleanAttribute={true}
  stringAttribute="string"
  intAttribute=1
  forcedOptional=?{Some("hello")}
  onClick={reduce(handleClick)}>
  <div> {ReasonReact.stringToElement("hello")} </div>
</MyComponent>

Différences avec le JSX de JavaScript

  • Les attributs et enfants ne nécessitent pas l’utilisation de {}, mais nous les montrons tout de même pour faciliter l’apprentissage. Une fois que vous refmttez votre fichier, certains d’entre deux s’en vont et d’autres deviennent des parenthèses.
  • Le prop spread de JSX n’est pas supporté : <Foo {...bar}>. Encore que dans le même ordre d'idées, nous avons le children spread, présenté plus haut : <Foo> ...baz </Foo>.
  • Le punning !

Punning

Le « punning » désigne le raccourci de syntaxe pour quand un libellé et sa valeur sont les mêmes. Par exemple, en JavaScript, au lieu de faire return { name: name }, vous pouvez faire return { name }.

Le JSX de Reason supporte le punning. <input checked /> n’est qu'un raccourci pour <input checked=checked />. Le formateur vous aidera à formater votre code ainsi dès que possible. C’est pratique dans les cas où il y a un grand nombre de propriétés à passer :

<MyComponent isLoading text onClick />

Par conséquent, un composant JSX en Reason peut se permettre d’avoir un peu plus de propriétés avant d’avoir recours à une bibliothèque évitant de passer des propriétés.

Note : ceci est une différence avec le JSX de ReactJS, qui ne permet pas le punning. Avec ReactJS, <input checked /> devient <input checked=true />, afin de se conformer aux idiomes de DOM pour compatibilité.

Conseils & astuces

Pour les auteurs de librairies souhaitant profiter du JSX : l'attribut [@JSX] ci-dessus est un hook pour les macros ppx potentielles pour localiser une fonction souhaitant se formater en JSX. Une fois que vous trouvez la fonction, vous pouvez la transformer en toute autre expression.

De cette façon, tout le monde bénéficie de la syntaxe JSX sans avoir besoin d'opter pour une librairie spécifique en l'utilisant, ex : ReasonReact.

Les appels JSX prennent en charge les fonctionnalités des fonctions labellisées : facultatif, explicitement passé en option et facultatif par défaut.

Décisions de conception

La façon dont nous avons conçu ce JSX est liée à la façon dont nous aimerions aider le langage à évoluer. Voir la section "What's the point?" dans cet article.

La possibilité d'avoir des macros dans le langage + la syntaxe agnostique de JSX permet à chaque librairie d'implémenter potentiellement JSX sans tracas. De cette façon, nous ajoutons des connaissances visuelles au langage OCaml sous-jacent sans compromettre ses sémantiques (aka, comment il s'exécute). L'un des principaux objectifs de Reason est de permettre au plus grand nombre de personnes de profiter du beau langage qu'est OCaml, tout en rejetant les débats fastidieux autour de la syntaxe et du formatage.

* Vous pourriez vous demander pourquoi vous n'avez jamais eu besoin de ces enfants dans ReactJS. ReactJS utilise une logique d'exécution spéciale + des transformations syntaxiques spéciales + une détection et un marquage d'argument variadique pour éviter la plupart de ces cas (voir ici). Une telle utilisation dynamique complexifie un peu la détection du système de types. Puisque nous contrôlons toute la syntaxe et ReasonReact, nous avons décidé d'introduire le spread d'enfants pour enlever toute ambiguïté entre le cas du wrapping et celui du non wrapping, sans coût de compilation et d'exécution !
