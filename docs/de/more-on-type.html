Mehr über Typen

Typargument!

Typen können Parameter aufnehmen, ähnlich wie Generics in anderen Sprachen. Es ist als wäre ein Typ eine Funktion, die Argumente annimmt und einen neuen Typen zurückgibt! Die Parameter müssen mit ' beginnen.

Der Anwendungsfall eines parametrisierten Typen ist, Duplikate zu eliminieren. Vorher:

/* this is a tuple of 3 items, explained next */
type intCoordinates = (int, int, int);
type floatCoordinates = (float, float, float);

let buddy: intCoordinates = (10, 20, 20);

Nachher:

type coordinates('a) = ('a, 'a, 'a);

/* führe die coordinates-"Typfunktion" aus und geben den Typ (int, int, int) zurück */
type intCoordinatesAlias = coordinates(int);

let buddy: intCoordinatesAlias = (10, 20, 20);

/* oder schreibe dies, wie weiter verbreitet, in einer Zeile */
let buddy: coordinates(float) = (10.5, 20.5, 20.5);

In der Praxis werden Typen für dich abgeleitet. Die etwas prägnantere Version des obigen Beispiels wäre also nichts anderes als:

let buddy = (10, 20, 20);

Das Typsystem leitet her, dass dies ein (int, int, int) ist. Nichts weiteres müsste ausgeschrieben werden.

Typargumente tauchen überall auf.

/* wird als `list(string)` abgeleitet */
let greetings = ["hello", "world", "how are you"];

Wenn Typen keine Parameter akzeptieren würden (d. h. wenn wir keine „Typfunktionen“ hätten), dann müsste die Standardbibliothek die Typen listOfString, listOfInt, listOfTuplesOfInt, usw. definieren.

Typen können mehrere Argumente aufnehmen und sind kombinierbar.

type result('a, 'b) =
  | Ok('a)
  | Error('b);

type myPayload = {data: string};

type myPayloadResults('errorType) = list(result(myPayload, 'errorType));

let payloadResults: myPayloadResults(string) = [
  Ok({data: "hi"}),
  Ok({data: "bye"}),
  Error("Something wrong happened!")
];

Gegenseitig rekursive Typen

Genau wie Funktionen können Typen durch and gegenseitig rekursiv sein:

type student = {taughtBy: teacher}
and teacher = {students: list(student)};

Beachte dass am Ende der ersten Zeile kein Semikolon steht und dass in der zweiten Zeile kein type zu finden ist.

Designentscheidungen

Ein Typsystem, das Typargumente erlaubt, lässt grundsätzlich Funktionen auf Typenebene zu. list(int) ist eigentlich die list-Typfunktion, die einen int-Typen annimmt und den finalen, konkreten Typen zurückgibt, den du an manchen Stellen benutzen würdest. Du hast vielleicht bemerkt, dass dies in anderen Sprachen mehr oder weniger "Generics" genannt wird. Zum Beispiel ArrayList<Integer> in Java.

Das Prinzip der geringsten Macht gilt wenn du versuchst „Dinge zu erledigen“. Wenn der Problembereich es zulässt, wähle definitiv die am wenigsten abstrakte (also die konkreteste) verfügbare Lösung, so dass die Lösung schneller erreicht wird und weniger instabile Umwege hat, die du durchlaufen musst. Dies bedeutet, zum Beispiel, Typen gegenüber frei geformten Daten, datengetriebene Konfiguration gegenüber turingvollständigen Funktionsaufrufen, Funktionsaufrufe gegenüber Makros, Makros gegenüber Projektabspaltungen, usw. vorzuziehen. Wenn du deine Domäne und Macht einschränkst, dann wird es leichter die Dinge zu analysieren. Das heißt, falls die Domäne ausreichend eingeschränkt ist, dies zu erlauben.

Wenn ein Typsystem ein allumfassender Aspekt deines Programms ist, müssen wir sicherstellen, dass wir genug Kraft übrig lassen, um deine Ausdruckskraft nicht übermäßig einzuschränken. Ohne „Typfunktionen“ würdest du am Ende ziemlich viel Boilerplate-Code haben, z. B. hart kodierte listOfInt, listOfString, listOfArrayOfFloat, ihre entsprechende Hilfsfunktionen usw. Bitte stelle jedoch auch sicher, dass du die Macht, die dir durch ein ziemlich leistungsfähiges Typsystem gegeben wird, nicht übermäßig missbrauchst. Manchmal ist es in Ordnung, ein kleines bisschen Boilerplate-Code zu schreiben, um die Notwendigkeit für andere, besonders mächtige Typen zu reduzieren. Wenn überhaupt, werden dein Pragmatismus und Urteilsvermögen durch geschmackvolle Kompromisse deutlicher als durch raffinierte Typen!
