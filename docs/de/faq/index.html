Häufige Fragen

Sollte ich zuerst Reason oder OCaml lernen?

Es ist nicht nötig, zu wählen! Reason und OCaml teilen exakt die gleiche Semantik (d.h. wie der Code ausgeführt wird). Nur die Syntax unterscheidet sich. Nutze Reason-tools, sodass du frei zwischen den beiden Syntaxen konvertieren kannst. Ein Reason-Tutorial ist ein OCaml-Tutorial und umgekehrt. Im Terminal könntest du folgende Aliase verwenden:

# konvertiert ocaml code zu reason
alias mlre="pbpaste | refmt --parse ml --print re --interface false | pbcopy"
# konvertiert reason code zu ocaml
alias reml="pbpaste | refmt --parse re --print ml --interface false | pbcopy"

Sie nehmen deinen Code aus der Zwischenablage (macOS), konvertieren ihn und fügen ihn wieder in deine Zwischenablage ein! Tausche pbpaste / pbcopy mit den Zwischenablagefunktionen deines Systems aus.

Ich bin mir nicht sicher, was ich mit Reason tun soll

Wir kompilieren sehr gut nach JS. Überlege dir, was für ein Projekt du normalerweise umsetzt, wenn es reines JavaScript wäre. Versuche, es stattdessen in Reason + BuckleScript zu portieren / schreiben! Wir empfehlen, es eher mit konkreten Projekten für Endbenutzer zu probieren (z. b. ein kleines Kommandozeilenwerkzeug) als Infrastruktur-Projekte (z. B. ein Generator für Boilerplate-Code). Letztere Kategorie erfordert Expertise und Verständnis von idiomatischen Reason-Code.

Wie hängen Reason, BuckleScript und OCaml zusammen?

Siehe hier. Reason ist eine Syntax für OCaml und unterstützt alle seine Features. BuckleScript kompiliert OCaml-/Reason-Code nach JavaScript.

Wo kommen all diese print_endline, string_of_int Funktionen her?

Sie stammen aus der Standardbibliothek und sind bereits während der Kompilierung deiner Datei geöffnet. Daher liegen sie im Sichtbarkeitsbereich.

Du kannst mehr über die Pervasives Bibliothek in der API Dokumentation erfahren:

https://reasonml.github.io/api/Pervasives.html

Kann ich eine Funktion zur Ausgabe beliebiger Datenstrukturen haben?

Wenn du mit BuckleScript nach JavaScript kompilierst, kannst du JS console.log durch Js.log verwenden. Wenn du zu nativem Code kompilierst, dann kannst du ppx_show verwenden. Ein zukünftiges OCaml-Feature (modular implizit genannt) löst das direkt in der Sprache.

Warum gibt es ein +, um ints zu addieren, und +. für die Addition von floats usw.?

Siehe hier.

Funktioniert die Bibliothek ___ mit Reason?

Most JS libraries should easily work under Reason + BuckleScript. On the native side, since Reason's just a syntax transform: yes, they work with Reason too. But the native workflow is currently work-in-progress and needs polish.

Wie sieht es serverseitige aus? Sollte ich zu nativem Code oder zu JS kompilieren und node.js benutzen?

We do compile to native, but the native workflow is currently work-in-progress. At this time, we recommend compiling to JS through BuckleScript and use the JS wrappers at reasonml-community or somewhere else.

Wie sieht es mit asynchronem Code in BuckleScript aus?

First, if you're not interfacing with any library that uses promises, you can simply use callbacks. Everyone gets them and they're performant.

If you need to bind to a JS library that uses promises, or communicate with such library, you can use BS's Js.Promise. There's also potential to have some syntactic sugar in the future. In the long run, we'd like to implement a spec-compliant promises implementation in OCaml/Reason proper, so that the compiler optimizations could kick in.

For a more idiomatic OCaml solution: on the native OCaml side, we have lwt and Async. We don't use them in web right now, but we might in the future.

Wie sieht es mit (Unit) Tests aus?

Some of OCaml's language features (not just types) might be able to defer the need for unit testing until later. In the meantime, for compilation to JS, we're working on Jest wrapper. We'll look into using Jest for native too, if Jest is written using Reason in the future (no concrete plan yet). OUnit is a good, small native OCaml testing library right now.

Was ist die .merlin-Datei im Wurzelverzeichnis meines Projekts?

That's the metadata file for Merlin, the shared editor integration backend for autocomplete, jump-to-definition, etc. For the JavaScript Workflow, bsb the build system generates the .merlin for you; You don't need to check that into your version control and don't have to manually modify it.

Ich sehe keinen import oder require in meiner Datei. Wie funktioniert die Modulauflösung?

Reason/OCaml doesn't require you to write any import; modules being referred to in the file are automatically searched in the project. Specifically, a module Hello asks the compiler to look for the file hello.re or hello.ml (and their corresponding interface file, hello.rei or hello.mli, if available).

A module name is the file name, capitalized. It has to be unique per project; this abstracts away the file system and allows you to move files around without changing code.

Sind Some | None, contents, Array, List und all diese speziell? Wo kommen die her?

They're ordinary variants/records/module definitions that come with the standard library, opened by default during compilation out of convenience.

Was bedeutet ein Argument mit einem vorangestellten Unterstrich (z. B. _ oder _foo)?

Say you have List.map(item => 1, myList);. The argument item isn't used and will generate a compiler warning. Using _ => 1 instead indicates that you're intentionally receiving and ignoring the argument, therefore bypassing the warning. Alternatively, _item => 1 has the same effect, but indicates more descriptively what you're ignoring.

Was ist dieses MyModule.t, dass ich ständig sehe?

Assuming MyModule is a module's name, t is a community convention that indicates "the type that represents that module, whatever that means". For example, for the Js.String module, String.t is the type carried around and representing "a string".

Warum gibt es ein Js_promise und auch ein Js.Promise? Was ist mit Js_array, Js_string und was noch?

As a convention, Js_foo is the actual module, and Js.Foo is just an alias for it. They're equivalent. Prefer Js.Foo, because that's the official, public module name.

Wann werden modulare implizite & multicor & algebraische Effekte fertig sein?

They will one day. In the meantime, help us ship more Reason code! The popularity will help funnel more OCaml contributions. The less the OCaml folks need to worry about low-hanging fruits, the more they can focus on great research and execution!

Warum sind BuckleScript und bsb so schnell? Wie kann ich sie abbremsen?

BuckleScript is optimized for performance across the whole stack. You can try slowing it down by adding a dozen layers of indirections and metaprogramming. Try:

  • Ein paar Endlosschleifen hier und da einzufügen.
  • Ein JavaScript-Build-Werkzeug in die Pipeline reinzustopfen.
  • Mehr Abhängigkeiten für ein Hello World reinzuziehen.

Ich sehe, dass eine eigenartige .cmi/.cmx/.cmj/.cma Datei in einem Compiler-Fehler referenziert wird. Wo kommen diese Dateien her?

The OCaml community frequently uses file extensions to distinguish between types of source, artifacts, and metadata, depending on your build target (native/bytecode/JavaScript). The following is a overview of some of the file extensions you may come across:

Quelldateien

  • .ml: OCaml Quelldatei
  • .mli: OCaml Interface-Datei; legt fest, welche Teile der passenden ml-Datei für die Außenwelt sichtbar sind
  • .re: Reason-Quelldatei. Wie .ml, aber für Reason
  • .rei: Grund Interface-Datei. Wie .mli, aber für Reason

Kompilierte Dateien

  • .cmi: Kompilierte Interface-Datei (.rei/mli)
  • .cmx: Kompilierte Objektdatei für native Ausgabe (via ocamlopt)
  • .cmo: Kompilierte Objektdatei für Bytecode-Ausgabe
  • .cmj: Kompilierte Objektdatei für Web (via BuckleScript)
  • .cma: Bibliotheksdatei für Bytecode-Ausgabe (äquivalent zu .a-Dateien in C)
  • .cmxa: Bibliotheksdatei für native Ausgabe
  • .cmt: Enthält einen „Typedtree“ – grundsätzlich der AST mit der ganzen Info zu Typen
  • .cmti: Entspricht einer .cmt-Datei aber für Interface-Dateien
  • .cmxs: Dynamisch geladenes Plugin (für native Kompilierung)
  • .o: Kompilierte native Objektdatei
  • .out: Konventioneller Name / Erweiterung für die endgültige Ausgabe, die von ocamlc / ocamlopt erzeugt wurde (z. B. ocamlc -o myExecutable.out)

Andere OCaml-Dateien im Ökosystem

  • .mll: Definitionsdatei für lexikalische Analyse mit ocamllex
  • .mly: Definitionsdatei für Parsergenerator mit ocamlyacc
  • .mldylib: Enthält eine Liste von Modulpfaden, die zusammen kompiliert und archiviert werden, um ein entsprechendes .cmxs-Ziel (natives Plugin) zu erstellen
  • .mliv: Datei spezifisch für Batteries für gewisse benutzerdefinerte Vorverarbeitungen.
  • .mllib: OCaml Bibliothek (cma und cmxa)
  • .mlpack: OCaml Paket (cmo erzeugt mit dem -pack Flag)
  • .mlpp: Extlib-spezifische Dateien für gewisse benutzerdefinerte Vorverarbeitungen
  • .mltop: OCamlbuild top-level Datei, verwendet von OCamlbuild, um eine .top-Datei zu generieren
  • .odocl: OCaml Dokumentationsdatei

If some of those explanations are still a bit cryptic, here are expansions on some of the terms used above: - AST: Abstract Syntax Tree. The data structure coming from the source code, that the compiler operates on. - Linking: The step where the compiler takes many intermediate compiled files and assembles them together. E.g. linking A with B, because A's original source file referred to B. - Native: Builds that run on bare metal assembly instructions of the platform in question. - Bytecode: Like native code, but more portable and less performant - Object file: Contains machine code that is not directly runnable.

There is more information and context for many of these file extensions on the OCaml site and in this mailing list post. There are also deeper dives on native and bytecode compilation that contain more detailed descriptions in the OCaml manual.
