Record

Records ähneln JavaScript-Objekt aber sie sind

  • leichtgewichtiger
  • standardmäßig unveränderlich (immutable)
  • fest in Bezug auf Feldnamen und -typen
  • sehr schnell
  • etwas starrer typisiert

Verwendung

Typ (obligatorisch):

type person = {
  age: int,
  name: string
};

Wert (wird abgeleitet und hat hier den Typ person):

let me = {
  age: 5,
  name: "Big Reason"
};

Zugriff (die bekannte Punktnotation):

let name = me.name;

Record benötigt eine explizite Definition

Wenn du irgendwo zuvor nur {age: 5, name: "Baby Reason"} ohne eine explizite Deklaration schreibst, meldet dir das Typsystem einen Fehler. Wenn sich die Typdefinition in einer anderen Datei befindet, musst du explizit angeben, um welche Datei es sich handelt:

/* School.re */

type person = {age: int, name: string};

/* example.re */

let me: School.person = {age: 20, name: "Big Reason"};
/* or */
let me = School.{age: 20, name: "Big Reason"};
/* or */
let me = {School.age: 20, name: "Big Reason"};

Alle der obigen 3 ergeben "this record's definition is found in the School file". Die erste wird bevorzugt (die reguläre Typannotation).

Immutable Update

Neue Records können aus alten Records mit dem Spread-Operator ... erstellt werden. Der ursprüngliche Record ändert sich nicht.

let meNextYear = {...me, age: me.age + 1};

Dieses Update sehr effizient! Probiere ein paar in unseren Spielplatz, um zu sehen wie Records kompiliert werden.

Bemerkung: spread kann keine neuen Felder hinzufügen, da die Form des Records durch den Typ fest vorgegeben wird.

Mutable Update

Felder von Records können optional veränderbar sein. Dies gestattet dir, diese Felder an Ort und Stelle mit dem =-Operator zu ändern.

type person = {
  name: string,
  mutable age: int
};
let baby = {name: "Baby Reason", age: 5};
baby.age = baby.age + 1; /* alter `baby`. Happy birthday! */

Abgekürzte Syntax

Um Redundanz zu verringern stellen wir für die Typen und Werte eines Records sogenanntes punning zur Verfügung. Punning bezieht sich auf die abgekürzte Syntax, die du benutzen kannst, wenn der Name eines Felds mit dem Namen seines Werts/Typs übereinstimmt:

type horsePower = {power: int, metric: bool};

let metric = true;
let someHorsePower = {power: 10, metric};
/* same as the value {power: 10, metric: metric}; */

type car = {name: string, horsePower};
/* same as the type {name: string, horsePower: horsePower}; */

Beachte dass es kein Punning bei einem Record mit nur einem Feld gibt! {foo} tut nicht was du erwartest (es ist ein Block, der den Wert foo zurückgibt).

Tips & Tricks

Interoperabilität mit JavaScript

Wenn du mit JavaScript arbeitest, sollten sich die Record-Syntax & -Operationen vertraut anfühlen und du könntest versucht sein, mit JS zu interagieren, indem du ein JS-Objekt in einen Record konvertierst und umgekehrt. Das ist in Ordnung, aber wir haben einen noch besseren Weg ohne zusätzlichen Konvertierungsaufwand! See here

Record-Typen werden durch Feldnamen gefunden

With records, you cannot say "I'd like this function to take any record type, as long as they have the field age". The following works, but not as expected:

type person = {age: int, name: string};
type monster = {age: int, hasTentacles: bool};

let getAge = (entity) => entity.age;

The last line's function will infer that the parameter entity must be of type monster. The following code's last line fails:

let kraken = {age: 9999, hasTentacles: true};
let me = {age: 5, name: "Baby Reason"};

getAge(kraken);
getAge(me);

The type system will complain that me is a person, and that getAge only works on monster. If you need such capability, use Reason objects, described here.

Designentscheidungen

After reading the constraints in the previous sections, and if you're coming from a dynamic language background, you might be wondering why one would bother with record in the first place instead of straight using object, since the former needs explicit typing and doesn't allow different records with the same field name to be passed to the same function, etc.

  1. Die Wahrheit ist, dass tatsächlich in den meisten Fällen die Form deiner Daten in deiner App festgelegt ist, und wenn dies nicht der Fall ist, kann sie stattdessen möglicherweise besser als eine Kombination von Varianten (werden als nächstes vorgestellt) + Record dargestellt werden*.
  2. Da seine Felder fest sind, wird Record als Array kompiliert, mit Zugriff auf die Array-Indexe anstatt eines JS-Objekts (probier es im Spielplatz!). Auf nativer Ebene kompiliert es prinzipiell als ein Speicherbereich wobei der Feldzugriff einfach nur eine Feldsuche + ein eigentlicher Zugriff ist, auch bekannt als 2 Assembler-Anweisungen. Die gute alte Zeit, in der die Leute in Nanosekunden gemessen haben...
  3. Schließlich, da ein Record aufgelöst wird, indem diese einzelne explizite Typdeklaration gefunden wird (wir nennen dies „nominale Typisierung“), ergeben sich bessere Typfehlermeldungen als durch das Gegenstück („strukturelle Typisierung“, wie für Tupel). Dies erleichtert das Refactoring. Das Ändern der Felder eines Records ermöglicht dem Compiler auf natürliche Weise zu wissen, dass es immer noch derselbe Record ist, der bloß an einigen Stellen falsch eingesetzt wird. Andernfalls kann es bei der strukturellen Typisierung schwierig werden zu sagen, ob die Definitionsseite oder die Verwendungsstelle falsch ist.

* And we're not just finding excuses for ourselves! Reason objects do support these features.
